量化 agentic coding eval 中的基础设施噪声

原文：Quantifying infrastructure noise in agentic coding evaluations

导读

coding agent 的评测不像选择题。它要 checkout 仓库、安装依赖、运行测试、修改文件和提交 patch。任何基础设施不稳定都会污染最终分数，因此可靠 eval 必须把噪声显式建模。

核心观点

  • agentic coding eval 同时测模型、工具、sandbox、依赖、测试和判分脚本。
  • 同一个 patch 可能因 flaky test 或环境差异得到不同结果。
  • 基础设施噪声会掩盖小模型差异，也可能制造虚假 leaderboard 波动。
  • 失败应区分模型错误、工具错误、环境错误、测试错误和判分错误。
  • 重复运行、环境快照和端到端 trace 是量化噪声的基础。
  • 能否诊断基础设施问题，本身也是 coding agent 能力的一部分。

正文解读

噪声来自哪里

真实代码评测链路很长：仓库 checkout、依赖安装、shell 命令、沙箱限制、网络状态、时间限制、测试执行和判分脚本都会影响结果。任一环节不稳定，pass rate 都会混入非模型因素。

这类噪声比普通文本 benchmark 更难处理，因为执行代码就意味着依赖环境，而环境天然有不确定性。

为什么必须量化

如果两个模型只差一两个百分点，而评测环境自身波动也有类似幅度，那么直接比较就不可靠。更糟的是，优化测试策略或 retry 逻辑可能被误读为模型能力提升。

量化噪声不是为了消灭所有不确定性，而是让团队知道分数的置信范围：多少任务失败于安装，多少失败于 timeout，多少在重复运行中不稳定。

从 eval 反哺 agent 设计

基础设施问题不只是评测污染，也是真实开发的一部分。优秀 coding agent 应能识别 flaky test、重跑关键用例、缩小失败范围、检查依赖，并向用户说明不确定性。

因此 eval harness 一方面要控制环境噪声，另一方面也可以设计专门任务来评估 agent 对不可靠环境的诊断能力。

深度解读

这篇文章把 coding agent leaderboard 拉回工程现实：我们测的从来不是纯模型，而是模型加运行时的系统。一个 pass rate 背后可能包含网络、依赖、测试和 retry 策略。真正有用的 eval 应该像生产可观测性一样分层、可复现、可追踪，否则团队可能以为自己提升了模型，实际只是调顺了环境。

落地启示

  • 给 eval 失败建立 taxonomy，至少区分模型、工具、环境、测试和判分错误。
  • 对代表性任务做重复运行，观察方差而不是只看单次结果。
  • 固定容器、依赖和系统快照，降低不可解释波动。
  • 保存完整 trace，让每次失败能被复盘。
  • 把历史 flaky case 纳入回归集，防止环境问题被反复误判为模型问题。
