长时应用开发的 Harness 设计

原文：Harness design for long-running application development

导读

这篇文章把 agentic coding 的重点从“模型会不会写代码”推进到“系统能不能承载持续开发”。长时应用任务横跨多个文件、测试循环和架构判断，harness 必须帮助 agent 维护目标、状态、工具反馈与验证闭环。

核心观点

  • 长时开发不是一次 prompt，而是持续规划、执行、观察和修正的闭环。
  • harness 应承载工作流、状态和验证，而不是替模型写死所有路径。
  • 目标、计划、代码事实、执行日志和测试结果需要分层保存。
  • 工具接口要可恢复，避免一次命令失败就丢失上下文。
  • checkpoint、diff、测试和运行日志是长任务的事实来源。
  • 人类介入点应放在需求澄清、架构选择、权限提升和发布前验证。

正文解读

为什么应用开发需要 harness

真实应用开发会不断产生新信息：测试失败、依赖缺失、需求澄清、边界条件和代码风格约束。模型如果只依赖当前 context window，很容易在修复局部问题时忘记早期目标。

harness 的作用，是把这条长链路变成可观察、可恢复、可验证的系统。它不应该替模型做完所有决定，而应该给模型一个稳定工作台：能查事实、能动手、能验证、能回滚。

状态分层与 context 管理

长时任务里最危险的状态管理方式，是把所有东西都混在聊天历史里。用户目标变化慢，执行日志变化快，测试结果需要可追溯，开放问题需要被保留；这些信息应该分别管理。

compaction 也不该只压缩成泛泛摘要。更有价值的是保留为什么选择某条路径、哪些假设已验证、哪些命令失败过、哪些文件属于修改范围，以及下一步需要什么证据。

验证与恢复

没有测试命令、构建命令或手工验收步骤，agent 很容易把“代码看起来合理”误认为完成。长时 harness 应该在每个阶段提供局部验证，最终再做全局验收。

失败也应被设计进流程。命令超时、测试 flaky、权限不足和需求偏差都不应该让 agent 原地重复同一路径；checkpoint、rewind、resume 和清晰的错误分类能让探索成本可控。

深度解读

这篇文章的核心，是把 agentic coding 看成运行时工程，而不是 prompt 技巧。模型能力越强，越需要一个不抢方向盘但能维持秩序的外部结构。harness 把不可控的长程交互拆成目标、状态、工具、验证和恢复几个层次，使 agent 的聪明能持续发挥，而不是在上下文膨胀和工具失败中逐渐失焦。

落地启示

  • 给每个长时任务定义明确验收条件，而不是只等 agent 自报完成。
  • 把关键事实写入结构化 notes 或 session log，避免只依赖压缩后的聊天历史。
  • 为测试、截图、构建和 diff 设计一等工具，让验证成为默认路径。
  • 对删除、发布、权限提升等动作设置明确人类确认点。
  • 保留失败命令和失败原因，帮助后续恢复而不是重复踩坑。
