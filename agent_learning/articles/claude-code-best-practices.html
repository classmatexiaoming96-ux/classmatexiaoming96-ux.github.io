Claude Code：agentic coding 最佳实践

原文：Claude Code: Best practices for agentic coding

导读

Claude Code 不是聊天框，它能读文件、运行命令、编辑代码并自主推进问题。让它表现稳定的关键，是管理 context、验证路径和项目环境。

核心观点

  • context window 会填满，性能会随噪声增加而下降。
  • 强工作流是先探索、再计划、再编码。
  • prompt 应提供命令、日志、URL、路径、截图和期望行为等具体上下文。
  • CLAUDE.md 应记录项目约定、常用命令、测试说明和风格要求。
  • 大功能可先让 Claude 访谈用户并写 self-contained spec。
  • 长会话需要 early correction、clear、compaction、subagents 和 checkpoint。

正文解读

把 Claude Code 当成协作者

Claude Code 能实际操作代码库，因此输入不应只有一句需求。给它错误日志、复现命令、相关路径、预期行为和约束，能显著减少猜测。

最佳流程通常是探索、计划、实现、验证。先让它理解代码形状，再让它改动，会比直接要求写功能更稳。

环境和验证

项目应提供明确测试命令、构建命令、lint 命令和本地运行方式。没有验证路径，coding agent 很容易停在“看起来合理”。

CLAUDE.md 可以沉淀项目约定，但不应过度膨胀。它应该写常用命令、代码风格、测试策略和不该触碰的边界。

长会话管理

长 session 会积累噪声。用户应尽早纠偏，必要时清理上下文、创建 checkpoint、拆分 subagent 或让 Claude 总结当前状态。

对大功能，先生成 spec 再实现，可以把需求讨论和编码执行分开，降低反复修改造成的 context 污染。

深度解读

Claude Code 最佳实践的核心，是把 agentic coding 变成可验证协作，而不是把需求丢给模型许愿。模型很强，但它仍依赖项目事实、工具反馈和用户边界。好的使用方式像带一位高效新同事入组：给上下文、给测试、给约定、早纠偏、让它证明完成。

落地启示

  • 给 Claude Code 复现步骤、日志、路径和验收命令。
  • 先让它读代码和写计划，再批准实现。
  • 维护精简的 CLAUDE.md，不要堆无关规则。
  • 大任务先写 spec，小任务直接给明确边界。
  • 用测试、截图或脚本作为完成证据。
