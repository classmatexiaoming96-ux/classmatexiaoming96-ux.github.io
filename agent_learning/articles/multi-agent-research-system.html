我们如何构建多 agent 研究系统

原文：How we built our multi-agent research system

导读

开放式研究很难写死流程。问题会沿着资料不断分叉，早期搜索路径会影响最终结论。多 agent 架构用并行 subagents 扩展搜索广度和上下文容量。

核心观点

  • 系统采用 orchestrator-worker 架构，由 lead agent 协调专门 subagents。
  • lead agent 计划工作、创建 subagents、审查结果并决定是否继续研究。
  • citation agent 将最终报告声明映射回来源位置。
  • 多 agent 系统通常比 chat 交互使用更多 token，成本显著更高。
  • 它适合高价值、强并行、信息超出单一 context window 的任务。
  • 生产可靠性需要 checkpoints、tracing、resume 和确定性 retry。

正文解读

为什么研究适合多 agent

研究任务有路径依赖。一个线索会引出另一个线索，过早收敛会漏掉关键来源。单一 agent 容易受 context window 和搜索宽度限制。

多 agent 可以让不同 subagents 并行探索不同方向，再由 lead agent 综合判断。这相当于用额外 token 换搜索广度。

架构形态

lead agent 负责理解问题、拆分研究方向、维护 Memory、创建 subagents、审查结果，并判断是否需要继续搜索。

citation agent 则负责把最终报告中的 claim 映射回来源位置，降低长报告中引用漂移的风险。

成本与可靠性

多 agent 不适合所有任务。它会消耗远高于普通 chat 的 token，且容易出现过度 spawn、重复搜索、无限探索和工具选择不佳。

生产系统需要 checkpoint、trace、restart/resume、确定性 retry 和不破坏运行中 agent 的部署策略。

深度解读

多 agent 架构的本质，是用并行上下文和额外 token 扩展探索空间。它不是越多越好，而是适合那些价值高、信息分散、工具复杂、需要广度优先探索的任务。真正的工程难点不是 spawn subagents，而是控制何时分工、何时停止、如何合并证据、如何验证引用。

落地启示

  • 只在高价值、强并行任务上使用多 agent。
  • 让 lead agent 明确 delegation 策略和停止条件。
  • 为 subagents 提供不同搜索方向，避免重复工作。
  • 引入 citation pass，把最终 claim 对齐到来源。
  • 保留 trace 和 checkpoints，支持长任务恢复。
