原文：Introducing advanced tool use on the Claude Developer Platform

导读

本文介绍 Claude Developer Platform 的 advanced tool use：更灵活的工具调用、更清晰的 tool result 处理和更强 agent loop 能力，让开发者构建复杂工具型应用。

核心观点

  • advanced tool use 的目标是让 Claude 更可靠地与外部系统交互。
  • 工具不仅是函数调用，也是模型 context 和行动空间的一部分。
  • 复杂应用需要多工具选择、结构化参数、错误恢复和连续 tool loop。
  • tool result 应提供下一步决策所需信息，而不是倾倒全部后端字段。
  • 开发者需要设计工具描述、schema、权限和幂等性。
  • 平台能力越强，越需要 eval 和 trace 来理解 agent 行为。
  • advanced tool use 适合客服、数据分析、自动化、coding 和业务工作流。
  • 工具安全边界应包括输入验证、输出过滤、权限检查和审计日志。

正文

原文主线

本文介绍 Claude Developer Platform 的 advanced tool use：更灵活的工具调用、更清晰的 tool result 处理和更强 agent loop 能力，让开发者构建复杂工具型应用。

从 tool calling 到 tool use

早期工具调用常被理解为模型生成 JSON 参数、开发者执行函数、再把结果返回。advanced tool use 的重点更宽：模型要在多轮中选择工具、理解结果、处理错误、根据反馈继续行动，并在必要时改变计划。

这意味着工具不是附属 API，而是 agent 的行动界面。工具设计的质量会直接改变模型能力呈现。

工具描述和 schema

工具名称、description、参数名和返回格式都会进入模型 context。模糊 schema 会让模型填错参数，过长返回会污染上下文，不清晰错误会让模型重复失败。

好的工具应有明确用途、少量高信号参数、可预测返回和可操作错误。复杂后端 API 可以在工具内部聚合，给模型暴露更贴近任务语义的接口。

连续 tool loop

真实应用很少一次工具调用就结束。客服 agent 可能先查用户、再查订单、再检查政策、再执行退款；数据 agent 可能先取 schema、再运行查询、再画图、再解释异常。advanced tool use 要支持这种连续链路。

连续链路需要状态管理和错误恢复。工具失败时，模型应得到足够信息判断是改参数、换工具、重试还是向用户询问。

安全与可观测性

工具越强，越要有边界。外部写入、资金操作、权限变更、邮件发送等动作必须有权限控制、审计日志和必要的人类确认。模型输出不能直接等同于授权。

trace 是理解工具型 agent 的核心。开发者需要看到模型为什么选这个工具、传了什么参数、工具返回什么、下一步如何变化。没有 trace，advanced tool use 会变成黑箱自动化。

深度解读

advanced tool use 的关键，不是“模型会调用更多函数”，而是开发者平台开始把 agent loop 作为一等能力。函数调用只是语法，真正的产品能力来自工具语义、返回信息、权限边界和可观测性。未来很多软件接口会同时面对人类和 agent：人类需要 UI，agent 需要 ACI。谁能把工具设计成模型容易正确使用的形状，谁就能更早把 LLM 能力转化为业务自动化。

落地启示

  • 把工具设计当作产品接口设计，而不是简单后端函数暴露。
  • 返回结果要摘要化、可决策、可追溯。
  • 为写操作设计幂等性、权限检查和审计日志。
  • 记录完整 tool trace，支持失败复盘。
  • 用 eval 检查模型是否会在正确时机选正确工具。
