Claude Developer Platform 推出 advanced tool use

原文：Introducing advanced tool use on the Claude Developer Platform

导读

advanced tool use 的重点不只是让 Claude 生成函数参数，而是让模型在多轮中选择工具、理解结果、处理错误、调整计划并继续行动。

核心观点

  • 工具是 agent 的行动界面，也是 context 的一部分。
  • 复杂应用需要多工具选择、结构化参数、错误恢复和连续 loop。
  • tool result 应提供下一步决策所需信息，而不是倾倒全部后端字段。
  • 开发者需要设计工具描述、schema、权限和幂等性。
  • 平台能力越强，越需要 eval 和 trace 来理解 agent 行为。
  • 安全边界应包括输入验证、输出过滤、权限检查和审计日志。

正文解读

从 tool calling 到 tool use

早期工具调用常被理解为模型生成 JSON 参数、开发者执行函数、再把结果返回。advanced tool use 更像一个 agent loop：模型要持续观察工具反馈并决定下一步。

这意味着工具不再是附属 API，而是模型能力呈现的一部分。工具设计质量会直接决定 agent 是否可靠。

工具描述和 schema

工具名称、description、参数名和返回格式都会进入模型 context。模糊 schema 会让模型填错参数，过长返回会污染上下文，不清晰错误会诱导重复失败。

复杂后端 API 可以在工具内部聚合，对模型暴露更贴近任务语义的接口。好的工具应少参数、高信号、返回可操作错误。

安全与可观测性

工具越强，越需要边界。外部写入、资金操作、权限变更、邮件发送等动作不能因为模型输出而直接授权。

trace 是理解工具型 agent 的核心。没有模型选择、参数、返回和后续决策的完整记录，advanced tool use 会变成黑箱自动化。

深度解读

advanced tool use 的真正价值，是把 agent loop 作为平台能力，而不是把函数调用当成语法糖。未来很多软件接口会同时面对人类和 agent：人类需要 UI，agent 需要语义清楚、反馈紧凑、权限明确的工具面。谁能把工具设计成模型容易正确使用的形状，谁就能更早把 LLM 能力转成业务自动化。

落地启示

  • 把工具设计当作产品接口设计，而不是简单后端函数暴露。
  • 返回结果要摘要化、可决策、可追溯。
  • 为写操作设计幂等性、权限检查和审计日志。
  • 记录完整 tool trace，支持失败复盘。
  • 用 eval 检查模型是否会在正确时机选正确工具。
