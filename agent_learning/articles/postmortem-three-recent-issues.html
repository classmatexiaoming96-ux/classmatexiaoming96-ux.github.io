对最近三个问题的 postmortem

原文：A postmortem of three recent issues

导读

当用户说“AI 变笨了”时，问题未必来自模型本身。agent 产品质量可能被 prompt、context、tool policy、缓存、实验开关和基础设施共同影响。

核心观点

  • agent 产品问题通常不是单点 bug，而是多层变更叠加。
  • postmortem 应关注时间线、影响范围、根因、检测缺口和防复发措施。
  • 用户感知质量下降可能来自 context、prompt、tool policy、缓存或基础设施。
  • 多层系统需要分层指标，否则很难判断问题来自哪里。
  • 灰度、canary、回滚和 feature flag 是 agent 产品发布的关键安全阀。
  • 真实问题应反向更新 eval 回归集。

正文解读

为什么 agent postmortem 难

传统软件故障常能定位到代码路径、配置或依赖。agent 产品更复杂：模型版本、system prompt、tool definitions、context 管理、缓存、权限策略、UI 和 infra 都可能改变体验。

一个小 prompt 变更可能让模型更保守；一次 tool result 格式调整可能影响后续计划；缓存异常可能让长会话表现突然变差。

postmortem 应该记录什么

有用的 postmortem 要回答发生了什么、何时开始、影响哪些用户、如何检测、直接原因和深层原因分别是什么、为什么没有提前发现、如何防复发。

对 agent 系统，还应补充 trace：模型输入、tool calls、tool results、context 组织、prompt 版本、模型版本和实验开关。

防复发机制

防复发不能只靠以后小心。需要分层 metrics、canary、灰度、自动回归 eval、feature flags、快速回滚和更细 trace。

每次真实问题都应成为未来回归集的一部分，否则系统会反复踩同类坑。

深度解读

agent 产品 postmortem 的价值，是把模糊的“AI 质量下降”拆成可行动工程事实。模型行为是概率性的，但产品质量不能因此变成玄学。只有保存 trace、指标和发布历史，团队才能知道变化来自模型、prompt、tool、context 还是 infra。

落地启示

  • 为 agent 产品保存 prompt、模型、tool 和 context 版本。
  • 建立线上质量指标和用户感知指标。
  • 所有高影响变更走灰度和快速回滚。
  • 把 postmortem 发现加入 eval 回归集。
  • 公开复盘时区分已修复项和仍需监控项。
