引入 Contextual Retrieval

原文：Introducing Contextual Retrieval

导读

传统 RAG 把文档切成 chunks 后，很多文档级背景会丢失。Contextual Retrieval 的思路，是在索引前为每个 chunk 加上简短上下文说明，让检索更容易召回正确证据。

核心观点

  • 传统 RAG 切块会让 chunk 失去文档级背景。
  • Contextual Retrieval 为每个 chunk 添加简短上下文说明。
  • 上下文可由 Claude 预处理生成，并存入检索索引。
  • 方法可与 embedding search、BM25 和 reranking 结合。
  • contextual embeddings 与 contextual BM25 处理不同检索信号。
  • 上下文生成需要平衡信息量和 token/storage 成本。

正文解读

传统 RAG 的切块问题

chunk 离开原文档后，代词指代、章节主题、表格含义、术语定义和时间范围都可能丢失。检索系统可能因此错过正确片段。

生成阶段拿到缺乏上下文的 chunk，也可能被迫猜测原文含义，增加幻觉风险。

Contextual Retrieval 怎么做

方法是在索引前为每个 chunk 生成简短说明，描述它来自哪份文档、处于什么位置、讨论什么内容。

这些上下文与原 chunk 一起进入索引，使 embedding 和 BM25 都能利用额外背景。

组合与权衡

embedding 擅长语义相似，BM25 擅长关键词精确匹配。Contextual Retrieval 让两者都获得更完整文本，再用 reranker 进一步排序。

成本上，需要为每个 chunk 生成和存储额外上下文，因此应通过 retrieval eval 衡量收益，而不是盲目加长。

深度解读

Contextual Retrieval 的洞见很朴素：检索失败常常不是模型不懂，而是我们把材料切碎后抹掉了意义。它把一部分理解提前到索引阶段，用模型为数据加语境，再让传统检索受益。企业知识库的竞争力越来越像信息架构能力，而不是只看生成模型大小。

落地启示

  • 为 chunk 添加文档级上下文，而不是只保存原始片段。
  • 结合 BM25、embedding 和 reranking，而不是二选一。
  • 针对合同、财报、代码文档等类型设计不同上下文模板。
  • 用 retrieval eval 测召回率和排序提升。
  • 控制上下文长度，避免成本超过收益。
