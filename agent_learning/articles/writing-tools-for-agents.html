与 agents 一起为 agents 写工具

原文：Writing effective tools for agents — with agents

导读

这篇文章讨论如何为 agents 写工具，也如何让 Claude Code 等 agent 帮你优化工具。核心观点是：工具不是给人类开发者看的函数，而是给非确定性 agent 使用的行动界面。

核心观点

  • 工具是确定性系统与非确定性 agent 之间的新契约。
  • 工具质量会直接限制 agent 能完成任务的范围。
  • 应先原型化工具，再通过 agent eval 系统测量效果。
  • 工具命名和 namespace 应定义清晰功能边界。
  • 返回值要提供 meaningful context，而不是原始后端字段堆叠。
  • 工具描述、schema 和返回格式都需要 prompt engineering。

正文解读

什么是给 agent 的工具

传统函数面向确定性调用者：输入相同，输出相同。agent 则会在不确定上下文中决定是否调用工具、何时调用、传什么参数、如何解释结果。

因此工具设计不能只追求 API 完整性，而要追求 agent 可用性。一个对人类工程师清楚的函数，对模型未必清楚。

如何写工具

好的流程是先构建最小原型，让 agent 在真实任务中使用，再看失败轨迹：它是否选错工具、填错参数、误解返回、重复调用或忽略错误。

Claude Code 可以参与这个过程：生成 eval、分析 transcript、修改工具说明、压缩返回结果，并对比修改前后的成功率。

有效工具的原则

工具应选择正确粒度。太底层会迫使 agent 做过多编排，太高层会失去灵活性。namespace 要清楚，让模型理解边界。

返回结果要服务下一步决策。对 agent 来说，最有价值的不是全量字段，而是状态、关键证据、可选下一步和错误恢复信息。

深度解读

这篇文章把 tool design 提升为 agent 产品的核心能力。未来软件接口不只服务人类和程序，也服务会推理但会犯错的模型。工具越符合 agent 的认知形状，模型越能可靠行动。反过来，糟糕工具会把强模型变成一个不断猜参数、误读结果的脆弱系统。

落地启示

  • 用真实 agent 任务评估工具，而不是只看单元测试。
  • 让工具名称、namespace 和 description 清晰传达用途。
  • 把复杂后端细节隐藏在工具内部，返回任务相关语义。
  • 为工具建立 eval，持续优化成功率和 token 成本。
  • 让 Claude Code 帮忙分析 transcript 和改进工具说明。
