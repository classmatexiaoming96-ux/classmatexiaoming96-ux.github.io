MCP 代码执行：构建更高效的 agents

原文：Code execution with MCP: Building more efficient agents

导读

很多任务不该让模型在自然语言里硬算。MCP code execution 让 agent 在受控环境中运行代码，把数据处理、文件转换、统计验证和测试执行交给计算空间。

核心观点

  • 代码执行让 agent 不只阅读数据，还能计算和验证。
  • 把计算交给工具可减少 context token，避免模型手算。
  • 执行环境必须 sandboxed，限制文件、网络、资源和凭据访问。
  • 工具结果应返回摘要、关键输出和 artifact，而不是完整 stdout。
  • MCP 提供统一接口，让不同执行后端可被 agent 使用。
  • 代码执行与 context engineering 互补。

正文解读

为什么 agent 需要代码执行

处理大表格、比较日志、解析文件、运行统计、验证 API 返回和转换格式，都不适合让模型直接在 token 里完成。

更高效的方式是让模型写脚本、运行脚本、解释结果。agent 不必把数千行数据都读进 context，可以先过滤、聚合、抽样，再只读取高信号摘要。

MCP 的作用

MCP 把代码执行能力以统一工具形式暴露出来。后端可以是本地 sandbox、远程容器或专用数据环境，agent 只需要理解能力和约束。

这降低了集成成本，也允许企业把执行放在自己的安全边界内。

边界和返回值

代码执行是高风险能力。未经限制的执行环境可能读取敏感文件、访问网络、消耗资源或修改状态，因此要限制文件系统、网络、CPU、内存、时间和凭据。

返回值不应直接倾倒 stdout。好的结果包含 exit code、摘要、关键表格、错误类型、artifact 路径和下一步建议。

深度解读

MCP code execution 的长期意义，在于把 agent 从纯语言推理器变成可计算系统。很多所谓模型不擅长的任务，本来就不该让模型手算。关键不是给 agent 一台无限制机器，而是给它一套可审计、可限制、可恢复的计算工具。

落地启示

  • 把大数据处理放到执行环境里，只把高信号结果送回模型。
  • 按只读、临时写入、网络访问、持久化写入分层授权。
  • 限制执行时间、资源和文件访问范围。
  • 返回摘要和 artifact，而不是长 stdout。
  • 用 trace 记录代码、输入、输出和产物。
