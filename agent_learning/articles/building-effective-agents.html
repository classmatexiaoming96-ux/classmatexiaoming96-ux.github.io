构建有效的 agents

原文：Building effective agents

导读

Anthropic 的经验是，最成功的 agent 实现往往不是最复杂的框架，而是简单、可组合、可调试的模式。复杂度应随着任务需要逐步增加。

核心观点

  • agentic systems 可分为 workflow 与 agent：前者预定义路径，后者动态决定过程。
  • 构建 LLM 应用时应先找最简单可行方案。
  • agentic systems 常以延迟和成本换取任务表现。
  • 框架能降低入门成本，但也可能遮蔽 prompt 和 response，增加调试难度。
  • 常见模式包括 prompt chaining、routing、parallelization、orchestrator-worker 和 evaluator-optimizer。
  • 工具 prompt engineering 是 agent 成败的关键组成。

正文解读

什么时候用 agent

如果单次 LLM call 加检索和 examples 已经能解决问题，就不必引入完整 agent。agentic systems 往往增加延迟、成本和故障面。

workflow 适合路径明确、可控性强的任务；agent 适合需要灵活决策、工具选择和开放探索的任务。

简单模式优先

prompt chaining、routing、parallelization、orchestrator-worker、evaluator-optimizer 都是可组合 building blocks。团队可以先用少量代码实现，再在真实任务中验证。

框架有价值，但开发者需要理解底层 prompt、tool schema 和响应。否则框架抽象会让调试变难，并诱导过早复杂化。

工具和实践

工具应被当作 agent 的行动界面，而不是随手暴露的 API。名称、描述、参数、返回值和错误信息都会影响模型是否正确使用。

最终目标不是展示复杂架构，而是让系统在真实任务上可靠完成、可观察、可调试、可控制成本。

深度解读

这篇文章之所以长期重要，是因为它给 agent 热潮降温：有效 agent 不等于堆框架和多轮循环。好的系统从任务结构出发，先用简单 workflow 解决确定部分，再把需要动态决策的部分交给 agent。复杂度是成本，不是成就。

落地启示

  • 先验证单次调用或简单 workflow 是否足够。
  • 只在灵活性和开放探索确实带来收益时引入 agent。
  • 使用框架时仍要能看到 prompt、工具调用和响应。
  • 把 agent 模式当成 building blocks，而不是完整大框架。
  • 持续用真实任务 eval 成本、延迟和成功率。
