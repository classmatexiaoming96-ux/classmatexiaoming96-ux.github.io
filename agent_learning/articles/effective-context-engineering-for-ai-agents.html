面向 AI agents 的有效 context engineering

原文：Effective context engineering for AI agents

导读

prompt engineering 更关注模型被怎样指令，context engineering 更关注模型在某一刻能看到什么信息。对 agent 来说，context 是工作内存、证据包、工具结果和历史决策的组合。

核心观点

  • context engineering 与 prompt engineering 不同，重点是信息供给和组织。
  • agent 能力受限于可见 context 的相关性、时效性和密度。
  • 有效 context 应包含任务目标、约束、证据、工具状态和未解决问题。
  • 检索、压缩、memory、session log 和 agentic search 都是 context 工程手段。
  • 上下文过少会让模型猜，过多会稀释注意力和浪费 token。
  • 长时 agent 需要让 context 可恢复、可查询、可更新。

正文解读

context engineering vs prompt engineering

prompt engineering 常常关注一句指令怎么写。context engineering 关心的是，在模型生成下一步之前，它能看到哪些事实、哪些工具结果、哪些限制、哪些历史决策。

对 agent 来说，这个区别很关键。模型可能知道怎么推理，但如果拿到的是过期、噪声或缺失的上下文，就会做出错误行动。

有效 context 的解剖

高质量 context 通常包含用户目标、当前计划、已验证事实、未解决问题、工具状态、外部证据和可执行约束。它不只是更多文本，而是更合适的信息密度。

过多上下文会稀释注意力，过少上下文会迫使模型猜测。好的系统会按任务阶段动态选择信息，而不是把所有历史都塞进窗口。

检索和 agentic search

检索不是一次性动作。agent 需要在任务推进中不断提出新问题、寻找证据、更新假设，并把关键发现写回 memory 或 session。

这让 context engineering 与 harness 设计紧密相连：系统要决定什么被保存、什么被压缩、什么可重新查询，以及什么时候需要主动补证据。

深度解读

context engineering 的核心是承认模型不是直接面对世界，而是面对被系统组织过的信息界面。agent 产品的质量，很大一部分取决于这个界面是否把正确事实放在正确时刻。未来模型更强以后，context 仍然重要，因为更强推理会更依赖高质量证据，而不是更能忍受噪声。

落地启示

  • 把任务目标、约束、证据和开放问题分层管理。
  • 为长任务维护 session log，而不是只依赖当前窗口。
  • 用检索和 memory 提供可恢复上下文。
  • 压缩时保留决策原因和验证状态。
  • 通过 eval 测 context 策略，而不是只调 prompt。
