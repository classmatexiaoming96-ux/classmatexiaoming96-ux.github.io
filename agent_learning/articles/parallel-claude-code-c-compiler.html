用 Claude Code 团队构建 C 编译器

原文：Building a C compiler with a team of parallel Claude Codes

导读

C 编译器足够复杂：它需要语言规范、模块架构、边界条件和大量测试。把多个 Claude Code session 当成并行工程团队，是观察 agent 协作边界的好实验。

核心观点

  • C 编译器能考验 agent 的规范阅读、架构一致性和长程状态管理。
  • parallel sessions 可以分工实现 lexer、parser、type checker、codegen 和测试。
  • lead session 需要维护架构文档、共享 AST 约定和验收标准。
  • 并行开发的主要风险是接口漂移、重复实现和测试覆盖不一致。
  • 集成比单点实现更难，需要频繁 merge 和全量回归。
  • 人类仍应控制架构边界和最终验收。

正文解读

为什么是 C 编译器

编译器项目包含词法、语法、类型、诊断、IR、代码生成和运行时测试。每个子系统都可以独立推进，但整体必须共享同一套抽象。

C 语言还有历史包袱和大量边界情况，agent 不能只靠模板生成，必须持续查规范、构造测试和修复回归。

把 Claude Code 当成团队

parallel Claude Code 的核心收益是上下文隔离和并行探索。worker session 可以深挖一个模块，不必把全部细节塞给同一个 agent。

但并行不等于自由开工。lead session 需要承担技术负责人角色：拆任务、定义接口、维护设计文档、收敛实现，并决定什么时候集成。

接口与验证

编译器协作的关键是共享契约：token 类型、AST 节点、symbol table、type representation、diagnostic format 和 IR schema 都要尽早固定。

验证也必须分层。lexer golden tests、parser snapshot、type error cases、codegen smoke tests 和端到端执行测试共同构成集成安全网。

深度解读

这类实验最有价值的结论，不是 Claude 能否写出一个玩具编译器，而是多 agent 编程仍然需要组织设计。没有共享事实来源、接口契约和集成责任，并行 agent 会生成许多局部合理但整体冲突的代码。未来 agent 团队可能很强，但它们仍需要技术负责人、架构边界和持续验证。

落地启示

  • 给多 agent 项目指定 lead，负责设计文档和接口裁决。
  • 把共享 AST、错误格式和测试协议放在所有 worker 都能读取的位置。
  • 让 worker 小批量交付，避免长时间分叉后一次性合并。
  • 每个子模块都要求自带测试，lead 负责全量回归。
  • 并行前先固定“什么不能各自发明”。
