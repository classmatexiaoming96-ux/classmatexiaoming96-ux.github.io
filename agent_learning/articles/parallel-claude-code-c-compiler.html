用 Claude Code 团队构建 C 编译器

原文：Building a C compiler with a team of parallel Claude Codes

导读

核心观点

  • C 编译器是测试 agentic coding 的好任务，因为它需要语言规范、架构设计和大量验证。
  • 多个 Claude Code session 可像工程团队一样分工，但需要明确接口和集成节奏。
  • 并行开发的关键风险是接口漂移、重复实现和测试覆盖不一致。
  • lead session 应维护架构文档、任务拆分、共享 AST 约定和验收测试。
  • worker sessions 适合处理 lexer、parser、semantic analysis、codegen、runtime 和 test fixtures。
  • 集成比单点实现更难，需要频繁 merge、运行全量测试和处理冲突。
  • 编译器项目能暴露 agent 对规范阅读、边界条件和长程状态的管理能力。
  • 人类仍应控制架构边界和最终验收，而不是让 parallel agents 自发收敛。

正文

原文主线

本文把多个 Claude Code session 当成并行工程团队，分工实现 lexer、parser、type checker、codegen 和测试，展示 agent 协作在大型编译器任务中的组织方式。

为什么是 C 编译器

构建 C 编译器是一个很硬的工程任务。它包含词法分析、语法分析、类型系统、语义检查、IR、代码生成、错误信息、测试框架和大量边界情况。这个任务足够复杂，能考验 agent 是否只是会写局部代码，还是能在长期架构中保持一致。

C 语言也有大量历史包袱和未定义行为。agent 必须不断查规范、构造测试、修复回归，不能只靠模板式生成。

把 Claude Code 当成团队

parallel Claude Code 的思路，是让多个 session 分别承担子系统工作。一个 lead session 负责总体架构、接口和集成计划；其他 worker sessions 负责 lexer、parser、type checker、codegen、测试和文档等模块。

这种模式的收益是并行探索和上下文隔离。每个 worker 可以在自己的 context 中深入一个子问题，不会把所有细节塞给同一个 agent。代价是协调：接口必须清楚，否则每个 worker 都会发明自己的 AST、错误类型和测试习惯。

接口是协作核心

编译器的模块边界天然适合协作，但前提是共享契约明确。例如 token 类型、AST 节点、symbol table、type representation、diagnostic format、IR schema 都要尽早固定。lead session 应持续更新架构文档，让 worker 以同一事实来源工作。

如果缺少共享契约，parallel agents 会产生看似独立完成、实则难以集成的代码。最后人工或 lead agent 会被迫做大量胶水修复，抵消并行收益。

验证与集成

编译器开发尤其依赖测试。最小测试包括 lexer golden tests、parser snapshot、type error cases、codegen smoke tests、以及端到端编译运行。每个 worker 都应为自己的模块贡献测试，lead session 则负责全量回归。

集成节奏应该短。长时间让 sessions 各自开发，最后再合并，会放大接口漂移。更好的方式是小批量任务、频繁同步、持续跑测试，让冲突在早期暴露。

深度解读

这类实验最有价值的启示，不是 Claude 是否能写出一个玩具编译器，而是 parallel coding agents 需要怎样的组织结构。多 agent 编程不等于并行开工，真正难的是共享设计、接口契约和集成责任。换句话说，agent 团队仍然需要技术负责人。人类可以把更多局部实现交给 Claude，但必须设计信息流：谁维护规范、谁裁决接口、谁跑全量测试、谁处理回归。没有这些机制，并行只会制造更多不一致。

落地启示

  • 给多 agent 项目指定 lead，负责设计文档和接口裁决。
  • 把共享 AST、错误格式和测试协议放在所有 worker 都能读取的位置。
  • 让 worker 小批量交付，避免长时间分叉后一次性合并。
  • 每个子模块都要求自带测试，lead 负责全量回归。
  • 并行前先固定“什么不能各自发明”。
