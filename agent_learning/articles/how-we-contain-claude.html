我们如何在不同产品中约束 Claude

原文：How we contain Claude across products

导读

随着 Claude 进入聊天、编码、研究和工具执行场景，containment 不再是单一安全开关。不同产品有不同风险：读什么、能做什么、如何处理不可信输入、怎样记录和升级。

核心观点

  • 不同产品中的 Claude 面临不同风险类型和行动边界。
  • 防线通常由模型行为、harness policy 和系统隔离共同组成。
  • agent containment pattern 要限制文件、网络、工具、凭据和外部写入。
  • 不可信输入会影响模型，因此必须区分用户指令、工具结果和外部内容。
  • 可信度不是二元属性，而是随来源、权限和任务上下文变化。
  • containment 的目标是让 Claude 更有用，同时把失败后果限制在可承受范围。

正文解读

三类风险与三层防线

Claude 在不同产品里可能面临数据泄露、越权行动、被不可信内容诱导等风险。单靠模型“听话”不足以覆盖这些场景。

更稳的防线是多层组合：模型层遵循政策，harness 层限制工具和权限，系统层用 sandbox、network policy、vault 和审计日志限制后果。

contain agents 的模式

agent 越能行动，越需要明确边界。文件系统、网络、工具、凭据和外部写入都应分层控制，让低风险探索顺畅，高风险动作需要显式授权。

这类模式不是为了压低能力，而是为了让 agent 在默认安全边界内更自主。

信任 Claude 读到的内容

agent 会读取网页、代码、issue、邮件、文档和工具结果，其中很多内容不应被当作指令。系统必须区分来自用户的授权指令与来自外部内容的非可信文本。

prompt injection 的难点在于它利用语言界面混淆数据和指令。containment 需要让这种混淆即便发生，也难以触达敏感能力。

深度解读

这篇文章可以看作 Anthropic 对 agent 安全工程的总览。随着 Claude 从回答问题走向执行任务，安全不再是单点分类器，而是产品架构：模型、harness、工具、sandbox、凭据、审计共同定义行动空间。真正成熟的 containment 不是让 agent 什么都不能做，而是让它能在可解释、可恢复、可限制的范围内做更多事。

落地启示

  • 把风险按读、写、外联、凭据和外部影响分层。
  • 区分用户指令与外部内容，尤其是网页和工具返回。
  • 用 sandbox 和 vault 限制 prompt injection 的后果。
  • 为高风险工具加 policy、审批和审计。
  • 按产品场景设计 containment，而不是套一套通用规则。
