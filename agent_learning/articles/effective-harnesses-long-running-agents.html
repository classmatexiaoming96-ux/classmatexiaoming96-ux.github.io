长时 agents 的有效 harness

原文：Effective harnesses for long-running agents

导读

长时 agent 不只是一个 while loop。它要在几十到上百轮交互中保持目标、调用工具、读取结果、更新计划、处理失败，并在 context 被压缩后继续工作。

核心观点

  • session log 应作为事实来源，context window 只是当前工作视图。
  • memory 和 compaction 要保留可恢复信息，而不是泛泛总结。
  • tool loop 需要停止条件、预算和异常处理。
  • 权限边界应按风险分层，低风险自动化，高风险请求人类批准。
  • harness 中的模型能力假设会随模型升级过期。
  • 可观测性是生产长时 agent 的基本要求。

正文解读

harness 的角色

好的 harness 不是替模型做所有决定，而是给模型稳定边界。它定义哪些状态持久化、哪些工具可用、何时停止、何时请求人类、何时重试。

长时任务会持续制造熵：上下文膨胀、工具失败、目标漂移、状态丢失。harness 通过外部结构对抗这些熵。

session 与 context

session log 记录用户输入、模型计划、tool call、tool result、错误、checkpoint 和人类反馈。context window 则是从 session 中挑选出来的当前视图。

把两者区分开，才能在长任务中恢复和压缩。compaction 应优先保留目标、已验证事实、未解决问题、修改范围、失败尝试和下一步计划。

预算、停止和恢复

没有预算，agent 可能不停搜索、重复测试、反复修同一个错误。预算可以是时间、tool call、token、文件范围或 retry 次数。

完成也不应只看模型自述，而应满足验收条件。恢复能力则包括 checkpoint、resume、rewind 和 crash-safe session。

深度解读

这篇文章的底层思想，是承认模型智能需要运行时结构才能长期稳定发挥。agent 产品竞争不只是模型分数，而是 harness 能否把模型放进真实环境：让它能记住关键事实、受控使用工具、在失败时恢复，并在高风险点找到人。

落地启示

  • 把 session log 和 context window 作为两个不同概念设计。
  • 为 tool loop 设置预算和停止条件。
  • 把权限按风险分层，减少低风险打扰，收紧高风险动作。
  • 建立可观测 trace，而不是只看最终输出。
  • 模型升级后回头清理过期 harness 补丁。
