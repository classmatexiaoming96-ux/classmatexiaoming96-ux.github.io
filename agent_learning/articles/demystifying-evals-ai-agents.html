AI agents 的 eval 入门

原文：Demystifying evals for AI agents

导读

agent eval 不是给单轮回答打分。agent 会计划、调用工具、观察结果、修改环境并继续行动，真正的风险常常藏在中间轨迹里。

核心观点

  • 好的 eval 来自真实用户任务，并有明确成功标准。
  • 最终答案、工具轨迹、成本、延迟、失败恢复和安全性都应纳入指标。
  • 回归集应覆盖高频任务、边缘任务、历史失败和安全边界。
  • LLM judge 有用，但需要 rubric、校准集和人工抽检。
  • eval 应区分能力问题、工具问题、prompt 问题和产品约束问题。
  • 最小可用 eval 可以从几十个真实案例和人工判分开始。

正文解读

agent eval 为什么特殊

传统 LLM eval 常常是输入问题、输出答案、判定对错。agent eval 更难，因为它要评估行为过程：调用了什么工具、是否访问了正确账户、是否修改了不该动的文件。

最终答案正确并不代表过程安全。一个客服 agent 可能给出正确退款结论，却途中读取了错误用户数据；coding agent 可能测试通过，却改了不该改的公共接口。

从真实任务开始

最有效的 eval 来自用户请求、客服记录、开发工单、历史 bug 和失败案例。每个任务都需要明确什么算完成、哪些动作禁止、哪些信息必须引用。

早期不需要庞大数据集。几十个高质量案例，加上清晰 rubric 和人工复核，已经能让产品迭代从 impression 转向事实。

指标与 judge

agent 指标应包括 success rate、tool call 数、token 成本、延迟、错误恢复、权限请求和政策遵守。不同产品权重不同，研究、coding、客服各有核心风险。

LLM judge 能降低开放式评估成本，但必须配合 rubric、gold labels、校准和争议样本复查。高风险任务中，judge 只能做初筛。

深度解读

agent eval 的本质，是把“感觉它变好了”变成工程事实。失败可能来自模型、prompt、工具、权限、UI、环境或用户需求不清。没有 eval，团队只能围绕 demo impression 争论；有了 eval，才能知道每次修改改善了什么、牺牲了什么、是否引入回归。

落地启示

  • 先收集真实任务，不要凭空编 benchmark。
  • 为每个任务写成功标准和禁止动作。
  • 保存完整 trajectory，方便失败归因。
  • 把历史线上问题纳入回归集。
  • 把自动硬指标、LLM judge 和人工抽检组合起来。
