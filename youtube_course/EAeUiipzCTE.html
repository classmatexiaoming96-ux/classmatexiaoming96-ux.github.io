Microsoft Agent Framework 完全指南：Memory、Context 与 Function Tools

作者：Microsoft Agent Framework 教程 · 频道：AI Dev Masters · 时长：60 分钟 · ▶ 原视频

本文是中文整理版：基于 26 个字幕片段完整理解视频内容后重新组织叙事，包含实验代码演示和关键概念解析。

一、概述

本教程深入讲解了 Microsoft Agent Framework 的核心概念，涵盖三大主题：

  • Memory（记忆）：如何让 Agent 记住对话历史和关键信息
  • Context（上下文）：如何管理长对话、避免信息丢失
  • Function Tools（函数工具）：如何扩展 Agent 能力，让它与外部世界交互

整个教程以一个旅行规划 Agent（Travel Agent）为贯穿案例，从基础版本逐步扩展到多工具协调的复杂系统。

二、Memory 与上下文管理

2.1 为什么需要 Memory？

大型语言模型（LLM）本身是无状态的——每次对话都是独立的前缀，对话历史需要手动传入。Memory 的作用就是让 Agent 能够"记住"之前的交互内容：

  • 短期记忆：当前对话中的历史消息
  • 长期记忆：跨对话积累的用户偏好和知识

2.2 Context 压缩与摘要

当对话变得很长时，Context 窗口会塞满历史消息，导致：

  • 推理成本增加（更多 token = 更多费用）
  • 模型可能"迷失"在大量历史中，忽略重要信息

解决方案：Context 摘要（Summarization）

教程演示了一个摘要器 Agent（Summarizer Agent）的使用：

# 创建摘要器 agent
summarizer = SummarizerAgent()

# 对当前对话历史进行摘要
summary = summarizer.summarize(thread.messages)

# 用摘要创建新线程继续对话
new_thread = [Message(content=summary, role="user")]

这样可以将长对话压缩为精炼的摘要，保留关键信息同时降低 token 消耗。

2.3 消息限制与处理策略

每个 Agent 都有消息数量限制。当对话超过限制时，系统需要决策：

策略说明
截断直接丢弃最旧的消息（可能丢失重要上下文）
摘要用 AI 将历史压缩为摘要（推荐）
分层保留近期的原始对话， older 消息转为摘要

三、个性化上下文

3.1 什么是个性化上下文？

个性化上下文是指从对话中提取用户偏好，然后在后续对话中主动注入这些偏好，让 Agent 的回复更贴合用户需求。

3.2 实现机制

教程展示了一个完整的个性化流程：

# Step 1: 创建提取 prompt，从对话中提取用户偏好
extraction_prompt = """
从以下对话中提取用户的偏好设置，返回 JSON 格式：
{
  "preferred_language": "语言偏好",
  "travel_style": "旅行风格",
  "budget_range": "预算范围"
}
"""

# Step 2: 使用 LLM 提取偏好
preferences = llm.extract(extraction_prompt, messages)

# Step 3: 主动注入个性化消息
personalized_message = f"用户偏好：{preferences}"
thread.add_message(Message(content=personalized_message, role="system"))

3.3 实验验证

Task 4 专门对比了有无个性化上下文时 Agent 回复的差异：

  • 无个性化：Agent 回复通用、泛化
  • 有个性化：Agent 贴合用户偏好，如指定语言、预算、风格

这个对比实验让开发者直观看到个性化上下文的实际效果。

四、Function Tools 函数工具

4.1 工具的定义

在 Agent 语境下，工具（Tool）是 Agent 可以调用的函数、API 或外部服务，使其能够执行实际操作而不只是生成文本。

记忆 vs 工具：记忆帮助扩展 LLM 的知识；工具帮助扩展 LLM 的行动能力。

4.2 函数工具的组成

一个完整的函数工具包含：

属性说明重要性
name工具名称，Agent 通过它选择调用哪个工具高
description工具描述，Agent 根据描述决定是否调用极高
parameters参数定义，Agent 提取用户输入后传参高
return_type返回值类型中

4.3 天气 API 示例

教程演示了一个天气查询工具的完整实现：

# 工具定义
@function_tool
def get_weather(location: str) -> str:
    """
    获取指定位置的当前天气
    :param location: 城市名称
    :return: 格式化的天气信息
    """
    # Step 1: 调用地理编码 API 获取经纬度
    geo_response = requests.get(
        "https://geocoding-api.example.com",
        params={"location": location}
    )
    lat, lon = geo_response["latitude"], geo_response["longitude"]

    # Step 2: 用经纬度调用天气 API
    weather_response = requests.get(
        "https://weather-api.example.com",
        params={"lat": lat, "lon": lon}
    )

    # Step 3: 格式化返回
    return f"{location}: {weather_response['temp']}°C, 风速 {weather_response['wind']} km/h"

# 创建带工具的 Agent
agent = ChatAgent(
    tools=[get_weather]
)

# Agent 自动提取参数并调用
response = agent.chat("东京今天天气怎么样？")
# Agent 回复："东京今天气温 13.8°C，风速 4.1 km/h"

4.4 何时使用工具？

不是所有 Agent 都需要复杂工具。教程给出了清晰的决策指南：

场景推荐方案
简单 Q&A Bot基础对话 + 必要 Memory
RAG 场景知识库检索，无外部 API
需要实时数据天气、股票、新闻等 API
复杂业务流程多工具协调 + 子 Agent

五、实验与任务

Lab 3: 函数工具掌握

这个实验包含 4 个递进任务，完整覆盖函数工具的使用：

Task 1: 简单工具函数

定义一个计算器工具，支持加、减、乘、除运算。

@function_tool
def calculator(operation: str, a: float, b: float) -> float:
    """执行基本算术运算"""
    if operation == "add":
        return a + b
    elif operation == "subtract":
        return a - b
    elif operation == "multiply":
        return a * b
    elif operation == "divide":
        return a / b if b != 0 else "Error: division by zero"

agent = ChatAgent(tools=[calculator])
agent.chat("计算 50 * 20 = ?")

Task 2: 外部 API 工具

实现一个天气数据库查询工具，使用模拟数据：

weather_db = {
    "Paris": {"temp": "10°C", "condition": "晴", "humidity": "65%"},
    "London": {"temp": "8°C", "condition": "多云", "humidity": "80%"},
    "Tokyo": {"temp": "15°C", "condition": "阴", "humidity": "70%"}
}

@function_tool
def get_weather(city: str) -> str:
    """从天气数据库获取城市天气"""
    data = weather_db.get(city, {})
    return f"{city}: 温度{data['temp']}, 天气{data['condition']}, 湿度{data['humidity']}"

六、多工具 Agent

6.1 工具数组

当一个 Agent 需要多种能力时，可以传入工具数组：

agent = ChatAgent(tools=[
    search_web,        # 网页搜索
    send_email,        # 发送邮件
    calculator,        # 计算器
    get_weather       # 天气查询
])

6.2 多任务委托（Orchestration）

当 Agent 需要处理复杂任务时，可以启用多 Agent 协调：

# Agent A: 旅行规划 - 制定初步行程
planner_agent = ChatAgent(tools=[...])

# Agent B: 航班预订 - 专门处理机票
flight_agent = ChatAgent(tools=[book_flight])

# Agent C: 酒店预订 - 专门处理住宿
hotel_agent = ChatAgent(tools=[book_hotel])

# 协调流程：Planner → Flight Agent → Hotel Agent

这种模式叫做多任务委托（Multitask Delegation）或编排（Orchestration）。

6.3 实际案例

教程演示了 Agent 如何处理复合查询：

"帮我查询东京的天气，然后给我推荐三家五星级酒店。"

Agent 会自动：

  1. 调用 get_weather 获取东京天气
  2. 基于天气信息，调用 search_hotels 筛选酒店
  3. 整合结果返回给用户

七、生产级工具设计

7.1 错误处理与参数验证

Task 4 专门讲解生产级工具的必备特性：

Task 4: 健壮的参数验证

@function_tool
def query_user_database(user_id: int) -> str:
    """
    查询用户数据库
    :param user_id: 用户ID（必须是正整数）
    """
    # 验证 1: 类型检查
    if not isinstance(user_id, int):
        raise TypeError("user_id 必须是整数")

    # 验证 2: 范围检查
    if user_id <= 0:
        raise ValueError("user_id 必须是正整数")

    # 验证 3: 数据库存在性检查
    user = db.get(user_id)
    if not user:
        return f"Error: 用户 ID {user_id} 不存在"

    return f"用户: {user['name']}, 邮箱: {user['email']}, 角色: {user['role']}"

7.2 常见错误类型

错误类型示例处理方式
参数类型错误LLM 传入字符串而非整数显式类型转换 + 错误提示
参数范围错误传入负数 ID范围验证 + 友好错误信息
LLM 幻觉捏造不存在的参数值参数白名单 / API 实际查询验证
外部服务故障API 超时或返回错误重试机制 + 降级策略

7.3 何时不用工具

教程特别强调了不要过度设计：

  • 简单 Q&A Bot：只需要基础对话 + Memory，不需要工具
  • RAG 场景：知识库检索已经够用，无需额外 API
  • 工具会增加复杂度，应根据实际需求选用

八、术语解释

术语解释
Agent（智能体）能够自主理解目标、调用工具、执行任务的 AI 程序
Memory（记忆）让 Agent 记住对话历史和关键信息的机制
Context（上下文）当前对话的语境信息，包括历史消息和用户偏好
Context 窗口模型一次能处理的最大 token 数量
Summarization（摘要）将长对话压缩为精炼摘要的技术
Function Tool（函数工具）Agent 可调用的函数、API 或外部服务
Function Calling（函数调用）LLM 根据对话内容决定调用特定工具并传参
Orchestration（编排）多个 Agent 或工具协调工作的模式
RAG（检索增强生成）通过检索外部知识库来增强 LLM 回答质量的技术
Personalized Context（个性化上下文）从对话中提取并注入的用户偏好信息
