作者：David（Anthropic 技术员 / MCP 协议共同创建者） · 演讲场合：Code with Claude · 时长：约 26 分钟 · ▶ 原视频

本文是中文整理版：在视频内容的基础上重新组织、补图表、补对照表，方便快速阅读。关键节点附 YouTube 时间戳链接，可点击跳回原视频对应位置核对。

一、为什么要做这场分享

社区目前 1 万多个 MCP server，绝大多数只用了协议里最容易的那一块——工具（tools）。但 MCP 协议设计里其实有 5 个原语（3 个 server 端 + 2 个 client 端），合在一起能让 LLM 应用做出比"工具调用"丰富得多的交互（0:13）。

这次演讲想做三件事：

  1. 把 5 个原语过一遍，重点讲少有人用的那几个；
  2. 用一个聊天 App 例子演示怎么把 5 个原语组合起来；
  3. 谈 MCP 下一步——从本地走向 Web。

图 1：MCP 协议的 5 个原语——一边由 server 暴露给 client，一边由 client 暴露给 server。最关键的区别在于"谁来发起"。

原语 谁来发起 一句话定位 典型场景

三、Prompts：用户主动调入的"模板"

Prompts 是 MCP 里最被忽视的原语之一（0:51）。一句话：MCP server 可以暴露一组"提示模板"，让用户在 client 里主动选用。

flowchart LR Author[Server 作者] -->|定义| P1[Prompt:
summarize-pr] Author -->|定义| P2[Prompt:
review-issue] User[用户] -->|/summarize-pr| P1 P1 -->|参数自动补全
抓取 PR comments| Ctx[上下文窗口] Ctx --> Model[模型作答]

图 2：Prompt 的典型用法——用户敲一个 slash 命令，server 把对应模板（甚至动态抓取的真实数据）一次性塞进上下文。

它解决了什么问题

  • 你（作者）最懂自己的 server 该怎么用——把"招牌玩法"打包成 prompt，让用户开箱即上手；
  • prompts 是动态执行的——背后就是代码，不只是静态模板。可以在用户调用时去远端抓数据。

例子：把 PR comments 拉进上下文

David 在 Zed 编辑器里演示：用户敲 /pr-comments，prompt 背后就去 GitHub 拉 PR 评论，直接进上下文，然后用户跟模型说"按这些 review 意见把代码改一下"。

鲜为人知的加分项：Prompt Completion

定义 prompt 时还能给参数附补全函数。视频里用户敲完 /pr-comments 后，弹出可选 PR 列表供选择（3:23）。在 TypeScript 里实现这个需求只要几行——一个 completion 函数 + 一个生成 prompt 的函数，Claude Code + Claude 4 基本能直接给你写出来。

四、Resources：应用驱动的"原始数据接入"

Prompts 是给用户用的，Resources 是给应用（client）用的（4:42）。

flowchart LR Server[MCP Server] -->|expose| Res1[Resource:
db_schema/users] Server -->|expose| Res2[Resource:
db_schema/orders] Res1 --> App[Client 应用
Claude Desktop / VS Code] Res2 --> App App -->|方式一| Ctx[加入上下文窗口] App -->|方式二| Embed[做 embedding
RAG 召回] App -->|方式三| UI[像文件一样挂在边栏]

图 3：Resource 的三种消费方式——应用层决定怎么用，不只是"塞进上下文"这一种。

它解决了什么问题

  • 应用决定怎么用这份数据——上下文/索引/UI 渲染都行；
  • 是 MCP 中比较被低估的原语：很多人只想着"加进上下文"，但 client 完全可以拿来做 embedding/RAG。

例子：暴露数据库 schema

David 演示用 Astra DB 的 MCP server 把库表 schema 作为 resource 暴露。在 Claude Desktop 里像挂文件一样把表挂上，然后说"画一张 ER 图"，Claude 自己读 schema 自己画图（5:55）。

五、Tools：模型驱动的动作

第三个 server 端原语，大家都熟：模型在跑的过程中自己决定要不要调用，常见用法不展开。David 说"建过 MCP server 的人，第一次看到模型自己调用你建的工具——是最有魔力的时刻"（6:41）。

六、决定用哪一个：**Interaction Model（交互模型）**

3 个 server 原语功能上有重叠（都能往上下文里塞东西），到底用谁？协议规范里其实埋着答案，叫 interaction model（8:02）：

原语 由谁触发 同一份数据按"何时出现"决定用谁
Prompts 用户 主动 用户想立刻把它推到上下文里时
Resources 应用 决定 应用层有自己的策略——索引、按需召回、UI 渲染
Tools 模型 自主 模型在解决任务时按需调用

"应该让数据出现的时机和意图，决定你用哪个原语"——这是 David 给的核心判断准则。

七、Sampling：server 反过来让 client 跑一次模型

讲到 sampling 之前，David 抛了个具体问题（10:01）：

"我想给一个 issue tracker 做 MCP server，支持「总结某 issue 的讨论」。总结需要模型——但 server 又不知道 client 用的是哪个模型，更没有 client 的 API key。怎么办？"

最朴素的做法是 server 自己装 Anthropic SDK + 自己要个 key——立刻就尴尬： - 用户要在 server 上又配一把 API key； - 把"哪个模型、谁付钱、谁审隐私"的控制权从 client 抢走。

Sampling 的本质

让 server 反过来请求 client 帮忙做一次推理。

flowchart TD User[用户] --> Client[MCP Client
Claude Desktop] Client -->|tool 调用| Server[MCP Server
issue tracker] Server -->|sampling 请求
「帮我跑这段 prompt」| Client Client -->|"用 client 已配置的模型 + 订阅"| Model[模型] Model --> Client Client -->|completion 结果| Server Server -->|tool 结果| Client Client --> User

图 4：Sampling 让 server "借" client 的模型——server 不再需要自己持模型/key。

它带来的两个红利

  1. Client 始终掌握安全 / 隐私 / 成本——用户在 client 那里看清楚每次推理付了多少 token、走了哪个模型，不用再额外给 server 一把 key；
  2. 可递归 / 可链式——server A 调 server B，B 再要 sampling，请求一路冒泡到最上层的 client。任何一个 MCP server 想用模型，都向上要，永远是同一个 client 守门（12:17）。

David 坦言：sampling 是最让他兴奋、但目前 client 支持最差的原语。Anthropic 第一方产品今年内会落地（13:00）。

八、Roots：server 反过来问 client "你都开了哪些工作区"

第二个 client 端原语。命名 David 自己也吐槽"起得不好"（13:37）。

它解决什么

想象你做一个 git MCP server——希望它在 IDE 里跑 git 命令。问题：

server 怎么知道用户的 IDE 此刻开着哪些工程目录？

Roots 就是 client 暴露给 server 的"我现在打开了这些目录"接口，server 拿到后只在这些目录里干活，不乱跑。

VS Code 目前已经支持 roots（15:00）。

十、MCP 的下一步：从本地走向 Web

社区现在 1 万多个 server，绝大多数是本地的——Docker 容器、本地可执行。下一步：让 MCP server 成为网站——client 直接连 URL 就能用（17:08）。

要走这一步需要两块基础：

  1. Authorization（授权）——把私有数据安全地接入 LLM 应用；
  2. Scaling（可伸缩）——server 能像普通 Web API 一样横向扩展。

10.1 授权：OAuth 2.1

MCP 规范现在要求远程 server 走 OAuth 2.1。这个名字听着唬人，实际上就是 OAuth 2.0 + 业界已经在做的安全加固（18:44）。

flowchart LR User[用户] -->|输入 mcp.payment.com| Client[MCP Client] Client -->|OAuth 流程| Payment[payment.com
身份 + 授权] Payment -->|token| Client Client -->|带 token 调用| Server[mcp.payment.com
MCP Server] Server -->|用户私有数据| Client

图 6：远程 MCP 的两个典型部署——To-C 走"我去 payment.com 登录"那一套；To-B 走"公司 SSO（Okta / Azure AD）"。

场景 关键收益
公网产品 用户在 client 里直接配 mcp.payment.com，OAuth 登一下，不再需要信任随便一个开发者放在本地的 Docker 镜像——信任锚回到原服务
企业内部 现有 SSO / 身份提供方（Okta / Azure AD）继续生效，员工早上登一次，全天调任何内部 MCP server 都自动鉴权（21:17）

David 强调企业场景的好处："做 MCP server 的团队"和"做身份/集成的平台团队"可以清晰分工——这是大企业能跑出体量化部署的关键。

10.2 可伸缩：Streamable HTTP

新增的 Streamable HTTP 让 MCP server "像普通 REST API 一样水平扩展"（22:26）：

情形 行为
简单 tool 调用 直接 return application/json，连接关闭，下一次请求另一个 Lambda 接
需要流式 / 后续推送（sampling、notification 等） 升级到 stream，server 可以在最终 return 之前来回发消息

结果：MCP server 可以放在 Lambda / Cloud Run / 任何 serverless 上，跟普通 API 一样部署。

十二、一页速查

原语 方向 触发者 何时考虑
Prompts server → client 用户 想给 server 配一组"招牌玩法"或 slash 命令
Resources server → client 应用 想让 client 用更高级的方式（RAG / 索引）消费数据
Tools server → client 模型 经典动作调用
Sampling client → server server 反向 server 内部要用 LLM 但不该自己持模型/key
Roots client → server server 反向 server 需要知道 client 当前的 workspace 范围

一句话外带

大多数人把 MCP 当成"暴露工具的协议"，实际上 MCP 是一份关于"AI 应用 / 用户 / 模型三方协作"的完整协议——5 个原语恰好覆盖了 3 方所有方向的交互。要做出比工具调用更丰富的体验，先把 prompts / resources / sampling / roots 这四个被冷落的原语用起来。
