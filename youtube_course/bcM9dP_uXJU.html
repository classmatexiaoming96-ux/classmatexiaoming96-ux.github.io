一问题引入：为什么“会聊天”不等于“会工作”

这支视频的核心不是教你再背一套“AI Agent 定义”，而是把普通人最常见的 Claude 使用方式往前推一步：从问答、生成单个文件，推进到一个能接收目标、拆解步骤、生成文件并持续迭代的工作系统。作者开场的判断很直接：很多人仍把 Claude 当成高级搜索引擎，问一个问题、复制一段答案、再问下一个问题。这样当然有用，但它没有改变工作流本身。

真正的转折发生在你把 Claude 放进一个有项目记忆、文件结构和固定流程的 workspace 里。此时你交付给它的不是“帮我写一段内容”，而是“按我们约定的流程完成一个结果”。视频用两个例子说明这一点：一个研究报告 Agent，负责澄清范围、制定计划、研究、综合和保存报告；另一个内容再利用 Agent，负责把长脚本拆成短视频脚本和社媒素材包。

视频中最值得抓住的一句话可以转述为：你不再管理每一个微小步骤，而是设定目的地，让 Agent 按流程开过去。

二核心概念：三层 Claude 使用方式

作者把 AI 使用分成三层。第一层是 basic chat：你问问题，Claude 回答。第二层是 builder mode：你让 Claude 产出东西，比如脚本、邮件、总结或分析。第三层才是 agentic work：你把完整目标交给 Claude，它自己拆阶段、提问题、执行、检查并交付结果。

这个划分的价值在于帮你判断：当前任务到底只是需要一次回答，还是值得沉淀成可复用流程。许多低风险、可重复、有明确输入输出的任务，最适合从第二层升级到第三层。比如行业研究、内容改写、周报生成、竞品摘要、文档整理，它们都有可定义的开始条件、验收标准和产出格式。

三第一块地基：把 `CLAUDE.md` 当成 AI 的入职文档

视频反复强调，很多人跳过 `CLAUDE.md`，然后抱怨 Agent 输出泛泛而谈。原因很简单：没有项目上下文，Claude 每次都像第一天入职。`CLAUDE.md` 的作用是让 Claude 在每次进入 workspace 时都能读到“这个项目是什么、服务谁、偏好什么风格、文件该怎么放”。

作者给出的结构很朴素，但非常实用。先写项目背景，用两三句话说明 workspace 的用途；再写关于你的信息，包括受众、语气、专业背景和偏好；然后写规则，例如复杂任务先问澄清问题、执行前展示计划、文件名用小写短横线、输出不要填充废话；最后写目录结构，让 Claude 知道 workflows、outputs、resources 等文件夹分别放什么。

# Project Context
This workspace is used for AI assisted research, content creation,
and workflow automation.

# About Me
I create practical AI content for non-technical users.
Prefer concise, direct, actionable language.

# Rules
- Ask clarifying questions before complex tasks.
- Show a plan before execution.
- Save files with lowercase hyphenated names.

# Folder Structure
- workflows/: reusable instructions
- output/: completed deliverables
- resources/: reference material

关键不是这份文件写得多高级，而是它把反复口头说明的偏好变成项目默认值。对团队来说，这也是治理入口：同一个项目里的多次会话会继承相同标准，输出就不会在语气、文件命名和验收方式上漂移。

四第二块地基：workflow 文件比工具更重要

视频里有一个反直觉判断：对初中级 Agent 来说，最重要的组件不是复杂工具，而是 workflow 文件。workflow 文件本质上就是一份用自然语言写的标准作业程序，告诉 Claude 某类任务的目标、阶段、规则、输出格式，以及失败时该怎么处理。

这点很适合非程序员理解：你不是在写代码，而是在写流程。Claude 本身是协调者，它读 workflow，决定每一步做什么、什么时候问问题、什么时候暂停等你确认。工具集只是它能在 workspace 里读文件、写文件、整理目录、搜索文档和编辑文本的能力。流程不清，再多工具也只会把混乱自动化；流程清楚，基础工具就能完成很多工作。

可以把 workflow 文件理解成“给 AI 项目经理看的 SOP”。智能不在炫技集成里，而在清晰指令和可验收步骤里。

五实操步骤：搭一个研究报告 Agent

1. 不要先给题目，先设计系统

作者建议先打开 Claude Code workspace，不要立刻说“帮我研究某个话题”。你应该先说：我想设计一个研究 workflow。当我给你一个题目时，你先问我范围、受众和深度；然后形成书面计划；我批准后再研究；最后把结构化报告保存到 output 文件夹。

这一步的本质，是让 Claude 帮你写出未来可复用的流程，而不是只完成今天这一次任务。好的流程通常包含澄清阶段、研究阶段、综合阶段、写作阶段和保存前检查。确认结构后，再让 Claude 把它保存为类似 `research-agent.md` 的 workflow 文件。

2. 运行 workflow 时，先回答澄清问题

当你真正运行研究 Agent，例如要求它研究“2026 年 AI agents 的实际状态：哪些有效、哪些被过度吹捧、行业往哪里走”，设置正确的 Agent 不应该立刻开写，而应该先问问题。你需要回答受众是谁、语气如何、要多深、是否偏工具还是趋势、目标长度是多少。

3. 审查计划，再允许执行

Claude 随后会给出计划：要覆盖哪些部分、优先哪些资料、最终报告结构是什么。视频强调，不要跳过这个两分钟的计划审查。多阶段工作一旦跑偏，半成品很难修；执行前调整计划，成本低很多。

4. 让 Agent 完成并保存文件

计划通过后，Agent 才开始按阶段研究、综合、写作并保存。这个体验和普通聊天不同，因为你不是在一段一段要输出，而是在观察一个流程从输入走向结果。输出完成后，你还可以在同一个 workspace 里继续说“把执行摘要压到三点”或“在结尾补一个工具对比”，Claude 会基于已有文件局部修改，而不是从零再写一遍。

六第二个例子：内容再利用 Agent

研究 Agent 证明了“多阶段报告”可以被流程化，视频后半段又给了一个更贴近日常创作者的例子：repurposing agent。输入是一篇长脚本，Agent 要从中挑出三个最适合短视频的片段，为每个片段写一个 60 秒脚本，再生成 X thread、LinkedIn post、Instagram caption 等社媒素材，最后把结果导出成多个 PDF，放进 output 文件夹。

这个例子说明 Agent 不一定要连接十几个外部系统才有价值。只要输入清楚、输出格式固定、判断标准可写成规则，它就能节省重复劳动。更重要的是，它把“内容拆条”从临场手工判断变成可迭代流程：这周你可以先跑一个版本，下周根据真实发布效果更新 workflow 规则。

七五个常见错误：大多数 Agent 失败不是模型不够强

视频最后总结的避坑清单很实用。第一，不写 `CLAUDE.md`。没有持久上下文，Claude 不知道你是谁、要什么标准、文件该怎么组织，每次会话质量自然不稳定。第二，目标太模糊。“研究一下 AI”只会得到百科式泛泛内容；“研究非技术营销人员如何用 AI agents 做内容生产，聚焦免费工具，面向没写过代码的人”才会得到可用结果。

第三，跳过计划审查。多阶段 Agent 如果直接开跑，偏航成本会被放大。第四，不要求澄清问题。workflow 没写清“开始前先问问题”，Claude 就会自行假设，而假设通常是平庸输出的来源。第五，一次想搭太多系统。作者建议从一个低风险、反馈快、输入输出明确的 workflow 开始，跑可靠后再做下一个。

八决策矩阵：什么时候该做成 Agent？

不是所有任务都值得 Agent 化。一次性问题继续用聊天模式就好；单个明确产物可以用 builder mode；只有当任务会重复、步骤多、需要保存状态或后续迭代时，才值得写 `CLAUDE.md` 和 workflow 文件。

判断问题如果答案是“是”建议做法风险提醒
这个任务每周或每月都会重复吗？值得沉淀流程写 workflow 文件并定义输入输出流程不稳定前不要自动全放权
任务需要多个阶段吗？适合 Agentic work加入澄清、计划、执行、复查四段必须保留计划审批点
输出会继续被修改和扩展吗？需要持久 workspace保存到 output，并让 Claude 基于文件迭代文件命名和目录规则要提前写清
错误成本很高吗？先做半自动让 Agent 生成草稿和检查清单，由人批准不要在没验收前接入生产动作
