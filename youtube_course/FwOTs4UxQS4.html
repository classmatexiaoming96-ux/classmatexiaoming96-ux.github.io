AI Agent 到底是什么：从 LLM、工作流到自主迭代

英文原题：AI Agents, Clearly Explained

频道：Jeff Su 时长：10:09 播放量：4,626,178 主题：Agent 入门、RAG、ReAct、自动化工作流

一、问题引入：为什么“Agent”这个词总让人困惑

这个视频的价值不在于展示一个复杂框架，而在于把“AI Agent”从营销词汇拉回到一个可以判断的工程概念。Jeff Su 的切入点很朴素：很多解释要么直接进入工具调用、向量数据库和框架术语，要么只说“它能帮你做事”，但普通使用者真正需要的是一条从熟悉事物出发的理解路径。

视频把学习路线拆成三层：第一层是大型语言模型，也就是我们已经熟悉的 ChatGPT、Gemini、Claude 这类聊天应用背后的能力；第二层是 AI workflow，也就是人类预先写好路径、让模型按步骤调用工具；第三层才是 AI agent，也就是由模型承担“下一步怎么做”的决策工作。这个顺序很重要，因为如果跳过前两层，很容易把所有“接了工具的 LLM”都误叫成 Agent。

课程中的核心例子是“咖啡聊天”和“社交媒体内容生产”。当你问模型“帮我写一封约咖啡聊天的邮件”，这只是输入到输出；当你问“我下一次咖啡聊天是什么时候”，模型必须访问日历；当你继续问“那天天气怎么样”，模型又需要天气服务。这些例子让边界变得清楚：接入外部工具不等于 Agent，能不能自行判断该走哪条路径，才是关键。

二、核心概念解释：LLM、Workflow、Agent 的边界

1. LLM 是会生成文本的被动系统

大型语言模型最擅长的是根据输入生成输出。你给它一个 prompt，它根据训练数据、上下文和当前指令生成回答。它可以写邮件、改文案、总结文本、解释概念，但它默认不知道你的日历、公司内部文档、实时天气或某个私有数据库里的内容。

视频强调了两个特征：一是知识边界，模型不知道没有提供给它的私有信息；二是行为边界，模型通常是被动等待 prompt，然后给出回应。理解这两个边界，才能明白为什么 RAG、工具调用和 Agent 会出现：它们都试图让模型在回答前看到更多上下文，或在回答过程中执行更多动作。

2. AI Workflow 是人类预先设计的路径

当你写出一条固定逻辑：“如果用户问个人日程，就先查 Google Calendar，再把结果交给 LLM 回答”，这就是工作流。它可能包含很多步骤，比如从表格读取链接、用 Perplexity 抓取新闻摘要、交给 Claude 生成 LinkedIn 和 Instagram 文案，再每天早上定时运行。它看起来很智能，但关键路径仍然是人设计的。

工作流的优点是稳定、可解释、易测试。缺点也同样明显：它只能沿着事先设定的路径运行。你让它查日历，它就不会自然想到改查天气；你让它生成文案，它不会自动决定增加一个批评者模型来迭代质量。它能自动化步骤，但不负责重新设计步骤。

3. RAG 只是工作流中的一种常见形态

视频中特别把 RAG 降低到一个更容易理解的位置：Retrieval Augmented Generation 本质上就是“回答前先查东西”。这个“东西”可以是日历、天气、文档库、向量索引，也可以是搜索 API。RAG 并不是 Agent 的同义词，它更像是一个检索增强的工作流组件。

这个区分对工程实践很有用。如果你的任务只是“先检索，再回答”，用 RAG workflow 就足够；如果任务需要模型根据目标动态决定检索什么、调用什么工具、是否重试、是否修改计划，那才开始接近 Agent。

4. Agent 的变化点：LLM 接管决策

Jeff Su 给出的判断标准非常直接：要让一个 AI workflow 变成 AI agent，关键变化是“人类决策者被 LLM 替代”。也就是说，模型不只是执行步骤，而是要根据目标推理出步骤、选择工具、观察中间结果，并决定是否继续迭代。

在社交媒体例子里，人类原本要决定先收集新闻、再总结、再写帖文；还要判断输出不够好时是否修改提示词。Agent 版本则应该能自己推理：先收集链接比复制全文更高效，Google Sheets 比 Word 更适合结构化链接，生成完 V1 后可以增加一个评审步骤，让另一个模型按 LinkedIn 最佳实践批评并迭代。

三、关键论点与实操步骤：如何判断一个系统是不是 Agent

步骤一：先问“目标是谁给的，路径是谁定的”

如果用户给的是明确步骤，比如“打开表格，读取 A 列链接，逐条摘要，然后写成帖子”，这更像工作流。如果用户给的是目标，比如“每天早上根据今天值得关注的新闻生成适合 LinkedIn 的帖子”，而系统自己决定检索、筛选、写作、评估和重试方式，就更像 Agent。

这不是术语洁癖，而是直接影响系统设计。工作流适合强约束、可重复、路径稳定的任务；Agent 适合目标清楚但路径会变化的任务。把稳定任务硬做成 Agent，成本、延迟和不可预测性都会上升；把动态任务硬写成工作流，又会在例外场景里频繁断掉。

步骤二：确认工具调用是不是由模型选择

工具调用本身不是 Agent 的充分条件。一个系统可以固定地每次都查日历、查天气、生成语音，这仍然是 workflow。更关键的问题是：模型是否能判断“这次应该查日历还是天气 API，还是先问用户补充地点”？如果答案是 yes，系统才有了 Agent 的味道。

工程上可以把这个差异翻译成控制逻辑的位置：控制逻辑写死在代码里，就是工作流；控制逻辑由模型在运行时根据目标和观察结果生成，就是 Agent。不过真实系统常常是混合体：外层有安全边界和固定流程，内部某些节点允许模型自主规划。

步骤三：看它是否会观察结果并迭代

视频提到的第三个特征是迭代能力。一个 Agent 不应该只执行一次计划并返回，它还应能观察中间结果，判断是否达标，然后决定继续修改、换工具、增加检查步骤或停止。比如生成 LinkedIn 帖文后，Agent 可以调用另一个模型做风格批评，再根据批评重写，直到满足标准。

这也解释了为什么 Agent 系统通常需要状态管理、日志、预算限制和退出条件。没有这些工程护栏，所谓“自主迭代”很容易变成无限循环、重复调用工具或产出不可验证的内容。

步骤四：用 ReAct 理解“推理 + 行动”循环

视频用 ReAct 解释 Agent 常见配置：Reason and Act。模型先推理下一步应该做什么，再通过工具采取行动，然后观察行动结果，继续推理。这个循环并不神秘，它只是把“想一想”和“动手做”交替组织起来。

对使用者来说，ReAct 的意义是降低想象门槛：Agent 不是一个魔法黑盒，而是一个会在目标、工具和反馈之间循环的系统。对开发者来说，它提醒我们要定义可用工具、工具输入输出、错误处理、观察格式和停止规则。

五、引用块：视频中最关键的判断

判断一个系统是不是 AI Agent，关键不是它有没有接工具，而是 LLM 是否取代人类成为工作流里的决策者。

这句话可以作为一个实用测试。很多产品会把“自动化 + LLM + API”包装成 Agent，但如果所有路径都已经由人写死，模型只是按固定脚本填空，那它仍然是 workflow。反过来，如果模型能根据目标决定使用哪些工具、如何检查中间结果、什么时候重试或停止，那就进入了 Agent 的范畴。

这也解释了为什么“Agent 能力”需要和权限设计一起讨论。让模型成为决策者，意味着它可能选择访问更多工具、产生更多副作用、消耗更多资源。真正成熟的 Agent 系统不是只提高自主性，还要把自主性放进可观测、可审计、可回滚的边界里。
