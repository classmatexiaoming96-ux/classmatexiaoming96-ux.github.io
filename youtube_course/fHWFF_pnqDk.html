生产环境的 Vibe Coding：怎么放手又不翻车

作者：Eric（Anthropic 编码 Agent 研究员 / 《Building Effective Agents》合著者） · 演讲场合：Code with Claude · 时长：约 31 分钟 · ▶ 原视频

本文是中文整理版：在视频内容基础上重新组织、补图表、补对照表，方便快速阅读。关键节点附 YouTube 时间戳链接，可点击跳回原视频对应位置核对。

引子：作者为什么对这件事这么较真

Eric 在演讲开头讲了个细节（0:24）：去年他骑车上班把手摔骨折了，整整两个月没法自己敲代码——所有代码都是 Claude 写的。"如何让 Agent 真的能替我把活干完"对他来说不是抽象问题，是当时的吃饭问题。这次演讲就是他这两年的方法论提炼。

一、什么才算 vibe coding？

误区："只要大部分代码是 AI 生成的，就是 vibe coding。"

Eric 不同意这个定义。 用 Cursor / Copilot 把代码大部分让 AI 写——你还在紧反馈循环里：每段都看、每段都改。这不是 vibe coding。

他坚持引用 Andrej Karpathy 的原始定义（1:24）：

"Fully give into the vibes, embrace exponentials, and forget that the code even exists."

关键短语：forget that the code even exists——忘掉代码本身的存在。

紧反馈循环] B -->|进一步放手| C[Vibe Coding
不再读每行代码] C -.风险.-> D[低 stake:
个人小项目

生产系统]

图 1：从紧反馈到放手的光谱——目前社会上跑出"vibe coding"概念的成功案例，大多停留在 D 这一档；这次演讲的核心问题就是：能不能往 E 走？

为什么 vibe coding 火，又为什么风评两极

  • 正面：让非工程师也能造软件——这是 Cursor / Copilot 都没能解锁的人群；
  • 负面：第一次写代码的人造出来的东西全是漏洞——API key 泄露、订阅绕过、数据库里乱写垃圾数据……

所以"vibe coding 在 prod 用"听上去像伪命题，但 Eric 说：我们必须严肃想这个事。

图 2：AI 能独立完成的任务长度大约每 7 个月翻一倍（3:26）。

Eric 的论点（3:46）：

今天 Claude 一次能做 1 小时的活——你可以全看完，没问题。

但下一年它能产出一整天的工作量；再下一年一整周。你不可能用紧反馈方式跟得上。

那时如果你还坚持「不把每行代码都过一遍我绝不放手」——你就成了瓶颈。

编译器类比

Eric 给的最好类比（4:22）：

早期编译器刚出来的时候，工程师们也不信任它，会去读编译器生成的汇编，确认是不是和他们自己写的一样。这个习惯没办法 scale。

后来大家放弃了。今天你我都还知道有汇编这回事，但我们不读它，照样能把软件做好。

软件工程在未来若干年要重复这个跨越。

但要小心定义：忘掉代码，不能忘掉产品

这是 Eric 最核心的一句话（4:41）：

"Forget that the code exists, but not that the product exists."

把"产品行为是否正确"这件事接得住——这是放手 vibe coding 的前提。

三、怎么管理你不懂的东西？老问题了

Eric 提醒大家："放手让别人/某个系统替你干活"这件事，人类社会练了几千年（5:20）。

谁 管什么自己不亲手做 怎么管
CTO 某专业领域的实现细节 验收测试——不读代码也能验是否过线
PM 工程团队怎么写代码 试用产品——直接当用户用，看体验对不对
CEO 财务总账 抽查关键数字 + 切片——对几个能验证的数自己心里有底

结论：管理"自己不懂深处的事情"是几千年来管理学的常态。只有软件工程师特别不习惯——因为我们到目前为止都是纯 IC（个人贡献者），习惯把整栈嚼透。做生产级 vibe coding 的第一步，就是放下这个执念。

四、有一件事今天**还不能**靠抽象层验证：技术债

Eric 诚实承认（7:18）：

几乎所有"我不懂细节但能验证"的领域（财务、产品、运维），都能找到验收手段。

但技术债不行。 想知道一段代码会不会把未来扩展卡死、有没有埋暗坑——目前唯一靠谱的方法还是有人读这段代码。

怎么办？答案是只在"不重要"的代码上放手 vibe code。

六、姿势 1：**Be Claude's PM**——做 Claude 的产品经理

Eric 的主梗（9:56）：

"Ask not what Claude can do for you—ask what you can do for Claude."

要把自己当成 Claude 的 PM 而不是 Claude 的程序员。

给 Claude 的"入职 onboarding"

类比：一个工程师入职第一天，你直接说"实现某 feature"——他不可能交得出来。你得：

  1. 带他过一遍代码库；
  2. 给他需求 / 约束；
  3. 告诉他风格、模式、已有同类例子。

给 Claude 也一样。

flowchart LR Idea[一个 feature 想法] --> Explore[让 Claude 探索代码库] Explore --> Plan[让 Claude 跟你
共同制定 plan] Plan --> Doc[plan 落成 artifact] Doc --> Exec[新会话里执行
一次写完] Exec --> Test[跑测试 / 看产品]

图 5：Eric 的实际工作流——往往花 15~20 分钟准备 prompt，但准备完之后 Claude 一次性写对的概率非常高（10:35）。

谁不适合做生产级 vibe coding

Eric 直说（11:43）：

完全不懂技术、不能问对问题的人，不应该全自动 vibe coding 一个生产业务。他们当不了 Claude 的 PM，整个流程就会失控。

这不是劝退非技术人员——他们可以做游戏、做 demo、做艺术品；但真正上线服务、有用户数据的系统，需要会问对问题的人在旁边把舵。

Eric 拿出了 GitHub 上的截图——一次 22k 行的真实 PR，合进了 Anthropic 生产环境的强化学习代码库（12:17）。这不是单次 prompt 出来的，他们用了一套组合拳：

步骤 做了什么 命中本演讲的哪条原则

5. 设计可观察的输入输出 整个系统在边界处就能验证对错 不读代码也能验

结果不只是"省时间"

Eric 说真正震撼他们团队的是心智解放（14:04）：

当一个原本两周的项目变成一天，我们意识到——之前那种"两周的项目"，现在我们会主动多做几个。

软件的边际成本下来了，能消费的软件多了。

这是 vibe coding 的真正回报：不是省人力，是让你敢做更大的事。

九、Q&A 精华

Q1：vibe coding 时代，新人怎么学？

担心：以前我们是在 syntax / 调库 / 连接组件的痛苦里学会编程——现在那些痛苦都被 AI 接走了，怎么成长？

Eric 的回答（17:06）：

  • 悲观面：你确实不会经历那种"硬核挣扎"，就像他大学教授当年抱怨"现在的程序员不会手写汇编了"；
  • 乐观面：你身边永远有一个 pair programmer 24/7 待命。看到不熟悉的库可以当场问 Claude："这是啥？为什么用它不用别的？" 学得比以前快得多。
  • 更高维度：以前一个架构决定要 2 年才能验对错，未来可能 6 个月就出结果——愿意学的人能在同样日历时间里多走 4 倍的循环。

会被淘汰的是懒人——不是新人。

Q2：plan 要写多详细？是不是要写完整 PRD？

Eric（18:50）：

  • 取决于你在乎什么——只在乎结果就只描述需求，了解代码库的话可以指定类、参考实例；
  • 别过度约束：模型在"留一定空间"时表现最好；
  • 不要标准化 PRD 模板——按"给 junior 工程师交代"的颗粒度即可。

Q3：安全 / 漏洞怎么办？

提问者举例：被报最严重漏洞的 10 个 vibe-coded App，连职业黑客都没出手就被攻破。

Eric（20:54）：

  • 回到第 1 条原则——Claude 的 PM 要懂得问对问题，知道哪里危险；
  • 自己的 22k 行案例之所以敢做，是因为那个系统完全离线，已经事先确认"出错也没有外部攻击面"；
  • 媒体里被攻破的那些 case，几乎都是"压根不该 vibe code 上线的人"在做生产业务；
  • 期待未来出现「可证明安全的脚手架」——核心安全敏感的部分（auth、支付）官方写好，用户只 vibe code 业务部分，类似 Claude Artifacts 那种"前端隔离环境"扩展到带后端。

Q4：TDD 怎么和 Claude 配？

提问者吐槽：Claude 容易"一起把实现 + 测试都写了"，测试也写得过度依赖实现。

Eric（23:09）的具体打法：

  • 明确告诉 Claude 测试格式："写 3 个端到端测试——1 个 happy path + 2 个 error case"；
  • 要 end-to-end，不要单元测试细节——这种测试人类可读、不被实现绑死；
  • vibe coding 时只读测试，不读实现——测试过你就放心。

Q5：怎么算"embrace exponentials"？

提问者引 Karpathy 原话：模型只会变得更好，但不会在每个维度都变好——是不是该有所保留？

Eric（25:44）：

重点不是"模型会变好"——而是变好得比你想象的快。

不是稳定增量，是斜率本身会一直上去。

引 Dario Amodei + Mike Krieger 的话："Machines of loving grace 不是科幻，是产品路线图。"

对照计算机历史：90 年代多几 KB RAM 是大事；今天我们以 TB 为单位——不是线性变好了几十倍，是对数级好了几百万倍。对未来 20 年的设想，要按这个尺度算。

Q6：你的具体工作流？多久 /compact 一次？

Eric（28:07）：

  • 终端开 Claude Code + 同时在 VS Code 里滚动 review diff；
  • 在自己作为人类工程师"想去吃午饭"的那种自然停顿点做 /compact；
  • 常见套路：先让 Claude 探索代码库、做 plan、把 plan 落到一个 markdown 文档；
  • 然后 /compact——把 100k token 的探索压成几千字摘要，再开干。

Q7：多个 Claude 并行？git worktree？

Eric（30:03）：

  • Claude Code 起项目 + Cursor 收尾："确切知道要改哪几行"时用 Cursor 比 Claude 更快；
  • 进新代码库时，先用 Claude Code 让它替你画地图："这个 codebase 里 auth 在哪？给我类名、文件路径"——构建心智模型再动手；
  • 多开 / worktree 他没具体推一种姿势——但补充："用文件做 Agent 之间的信息总线"是当前的现实最佳实践。

一句话外带

Vibe coding 在 prod 不是赌运气——它的核心配方是：做 Claude 的 PM + 只在 leaf 节点放手 + 设计可验证的抽象层 + 接受指数曲线。今天可以选择不放手，但别让"必须读每一行"的执念在两年后把你变成团队的瓶颈。
