一句话定位：给 coding agent 装一个“预先建好的代码地图”。当 agent 需要理解代码时，不再 grep 满仓库，而是向 CodeGraph 的 MCP 服务发一次结构化查询，一个调用拿到入口点、相关符号、调用关系和源码片段。作者强调一个关键设计哲学——抽取过程是确定性的，源自 AST 而非 LLM 总结，因此结果稳定、可复现、可进 git。

分层流水线架构

CodeGraph 的核心是一条清晰的单向流水线，src/index.ts 里的 CodeGraph 类把各层串起来，对外暴露 init / indexAll / sync / searchNodes / getCallers / getImpactRadius / buildContext 等 API：

↓
02

↓

↓

↓

数据模型：22 种节点 × 12 种边

图的表达力来自精心设计的类型系统（src/types.ts）。节点（NodeKind）覆盖 22 种符号——从 function/method/class 到 interface/trait/route/component；边（EdgeKind）覆盖 12 种关系：

src/types.ts
// 12 种关系，每条边还带 provenance：'tree-sitter' | 'scip' | 'heuristic'
export type EdgeKind =
  'contains' | 'calls' | 'imports' | 'exports'
  | 'extends' | 'implements' | 'references' | 'type_of'
  | 'returns' | 'instantiates' | 'overrides' | 'decorates';

interface Edge {
  source: string; target: string; kind: EdgeKind;
  line?: number; provenance: Provenance;  // 谁推断出这条边
}

抽取层：19+ 语言的 WASM 解析

CodeGraph 用 web-tree-sitter + tree-sitter-wasms 跑 WASM 语法，不依赖任何原生编译，因此“一个二进制到处跑”。它对工程细节的打磨令人印象深刻：

  • 按需加载语法：只为项目里真实出现的语言编译 WASM grammar，控制内存。
  • Worker 每 250 文件回收一次（WORKER_RECYCLE_INTERVAL）：因为 WASM 线性内存只增不减，唯一回收堆的办法就是杀掉 isolate 重建。
  • 1MB 文件上限 + 10s 解析超时：跳过压缩 bundle / vendored 大文件，避免撑爆 WASM 堆。
  • Git 感知发现文件：git ls-files --recurse-submodules，尊重 .gitignore 与子模块。
  • 对 Svelte / Vue / Liquid / Delphi 表单（.dfm）这类 tree-sitter 搞不定的格式，写了专用 extractor。

解析层：tree-sitter 搞不定的，才是真功夫

这是 CodeGraph 最有价值的部分。tree-sitter 只能告诉你“这里有个调用”，却无法跨文件解析“它到底调的是谁”，更无法理解框架的动态分发。ReferenceResolver 用一套分层策略来补这个洞：

  • ① 内置过滤：跳过 stdlib / 内建函数（JS 内建、React hooks、Python 内建、Go stdlib…）。
  • ② 框架模式：识别 14+ 框架（Express / NestJS / Django / Laravel / Rails / Spring / Gin / Axum / ASP.NET / Vapor / SvelteKit / Vue-Nuxt / Drupal…），把 URL 路由 route 节点 连到处理函数。这里用的是对“去注释后源码”的正则，而非完整 AST，省开销。
  • ③ import 解析：按语言扩展名顺序解析（TS 会试 .ts/.tsx/.d.ts/index.ts…），并读 tsconfig 的 paths 别名、Cargo workspace 成员。
  • ④ 名称匹配：精确名 → 限定名 → 文件路径三级匹配，用路径邻近度给候选打置信分。
  • ⑤ 回调合成（最妙）：见下。
回调合成 callback-synthesizer：tree-sitter 永远追不到 emitter.on('mount', handler) 和 emitter.emit('mount') 之间的隐式调用。CodeGraph 主动“合成”这些边——它识别注册器（/^on[A-Z]|subscribe|addListener/）与派发器（/emit|trigger|notify|dispatch/），按字段名/事件字符串配对，凭空补出 React 重渲染、EventEmitter、Vue/Svelte 组合式的动态调用边，并打上 provenance:'heuristic' 标注合成来源与注册位置。

存储层：node:sqlite + FTS5

CodeGraph 直接用 Node 内置的 node:sqlite（DatabaseSync，真正的 SQLite 编进了 Node），外面包了一层 better-sqlite3 形状的接口。因为自带 Node runtime，所以 WAL、FTS5 全平台可用。它的 PRAGMA 调优很“数据库工程师”：

src/db/index.ts — 连接调优
db.pragma('journal_mode = WAL');      // 读不阻塞写，MCP+git hook+CLI 可并发
db.pragma('synchronous = NORMAL');    // 配合 WAL 安全
db.pragma('cache_size = -64000');     // 64MB page cache
db.pragma('mmap_size = 268435456');   // 256MB 内存映射 I/O
db.pragma('busy_timeout = 5000');      // 锁等待 5s，骑过正常 sync

schema 设计同样克制：edges 表只建 (source,kind) 与 (target,kind) 两个复合索引，靠左前缀扫描覆盖正反向查询，故意不建多余的单列索引以减轻写入负担；unresolved_refs 表把引用解析延后到抽取之后，避免抽取阶段 N+1 查符号表；nodes_fts 用 FTS5 虚拟表 + 触发器自动同步全文索引。

服务层：10 个 MCP 工具，与自适应预算

CodeGraph 通过 stdio 暴露 10 个 MCP 工具给 agent，每个都为“一次调用解决一类问题”而设计：

🔎
codegraph_context

给定一个任务描述，一次返回入口点 + 相关符号 + 关系 + 源码——替代昂贵的探索 agent。

🧭
codegraph_trace

追踪 from→to 的调用路径，包含动态分发的跳点，附带 provenance / 注册位置元数据。

🗺️
codegraph_explore

一次性勘察多个相关符号，按文件分组返回源码，受自适应输出预算约束。

💥
codegraph_impact

分析改动某符号的“影响半径”——多步 BFS 找出所有受牵连的代码。

📞
codegraph_callers / callees

谁调用了它 / 它调用了谁，支持多层深度，秒级返回。

⚡
codegraph_search

FTS5 驱动的全文符号搜索，带 kind 加权与路径相关性打分。

一个被低估的细节：explore 工具的自适应预算会随项目规模缩放——小项目单次给 18K 字符、大项目（≥25k 文件）给到 38K，每行还带 cat -n 式行号前缀，让 agent 能直接引用 file:line 而无需回读文件。还有一条硬不变量：单文件字符上限在任何档位都不能下降——这是被一个真实回归逼出来的（excalidraw 的 415KB App.tsx 曾因上限太低只返回不到 1% 内容，迫使 agent 退回去 Read）。

增量同步与安装器

FileWatcher 用原生 OS 事件（FSEvents / inotify / ReadDirectoryChangesW）+ 防抖，让图随写随更新；在 WSL2 /mnt 与网络盘上自动降级为 git hook（post-commit / merge / checkout）。安装器（@clack/prompts 交互式）会自动探测并配置 Claude Code、Cursor、Codex、opencode、Hermes——比如给 Cursor 注入 --path 解决其子进程 cwd 错误的问题。

对 coding agent 生态的意义

CodeGraph 把“代码理解”从 每次会话临时探索 变成 一次索引、长期复用的基础设施。它选择 100% 本地 + 确定性抽取，意味着结果可复现、可进 git、无隐私顾虑——这让它有资格成为多种 agent 共享的“代码索引标准件”。它证明了一件事：agent 性能的下一个数量级，可能不在更大的模型，而在更好的检索基建。

Monorepo 三件套

核心哲学：确定性结构 × LLM 语义

语义批处理：用社区发现切批次

朴素的“每 20 个文件一批”会把逻辑耦合的模块拆散，丢失跨批的语义边。Understand-Anything 在 2026-05 的设计里引入了一个聪明做法（compute-batches.mjs）：对 import 图跑 Louvain 社区发现，把语义耦合的文件聚成一批（5–35 个/批），超大社区再用边介数启发式拆分；同时为每个文件构造 neighborMap（跨批邻居导出的符号），让 file-analyzer 能自信地下跨批边的判断。这同时解决了另一个现实问题——在输出受限的后端（如 Bedrock Opus 4096–8192 token）上，agent 会自检输出大小并切成 batch-i-part-k.json。

指纹 + 陈旧检测 = 增量分析

fingerprint.ts 为每个文件算出结构指纹（内容 hash + 函数/类/import 签名）。变更被分成三级：NONE（hash 相同，跳过）/ COSMETIC（内容变但签名没变，只更新语义）/ STRUCTURAL（签名变，全量重分析）。配合 git diff 找出改动文件，只重分析它们，再 merge 回旧图——200 文件的库改了 3 个文件，秒级更新而非分钟级。

搜索：模糊 + 语义双引擎

搜索走 fuse.js 做关键词模糊匹配（name/tags/summary 加权），同时有一个 SemanticSearchEngine 对预计算 embedding 做余弦相似度检索，Dashboard 里可在 fuzzy / semantic 间切换。一个值得称道的隐私细节：写盘前 sanitiseFilePaths 会把所有绝对路径转成项目相对路径，绝不把 /Users/alice/company/... 泄漏进可提交的 JSON。

可视化 Dashboard

前端用 React Flow 渲染图、Zustand 管 150+ 字段的状态、ELK（Eclipse Layout Kernel）做层次布局、Louvain 做视觉聚类、Prism 高亮源码。三种视图（structural / domain / knowledge）可切换；还有角色自适应 UI——根据你是“非技术 / 初级 / 资深”动态调整细节密度，以及暗金奢华主题（#0a0a0a + 金色 #d4a574 + DM Serif Display）与 6 国语言本地化。

业务域叠加层是它区别于一切纯静态工具的杀手锏：在结构图之上，它能再抽出“订单管理 → 创建订单 → 校验输入”这样的领域 / 流程 / 步骤三级业务层级，每个领域节点都能按 ID 下钻到真正实现它的函数和文件——把代码和业务逻辑第一次连了起来。

真正的护城河：七条治理流水线

普通插件市场（npm 之类）几乎没有审查。Anthropic 把“官方”做成了一套可审计、可自愈的 CI/CD 治理体系，跑在 .github/workflows/ 里：

  • validate-plugins：PR 时校验结构与 schema，对官方市场而言 SHA 锁定缺失 = 硬错误。
  • scan-plugins（核心）：用 Claude 本身按 policy/prompt.md + schema.json 对每个外部插件做安全策略扫描，输出结构化裁决：是否联网、是否下载软件、hook 是否“宽作用域”、是否有未披露的遥测、描述是否与行为相符。裁决带 30 天缓存（按策略 hash），策略一改全部失效。用 WIF 联合身份鉴权，无静态 API key。
  • bump-plugin-shas：每晚 07:23 UTC 自动追踪上游 HEAD，新 SHA 先 validate 再开一个汇总 PR，并触发扫描；单次最多 130 个，防成本失控。
  • revert-failed-bumps：若扫描发现某插件策略不过，自动把它的 SHA 回滚到主干值、重扫、在 PR 上评论违规项——一个坏上游不会卡住整批 bump。带 3 次/晚的熔断防循环。
  • check-mcp-urls：每日探活所有 http/sse MCP 端点，404/410/连接失败即报警。
  • validate-frontmatter：用 Bun + yaml 校验 agents/skills/commands 的 YAML frontmatter（含 glob 特殊字符处理）。

插件谱系：从 LSP 到 MCP

目录里 300+ 条目覆盖 development(92) / productivity(39) / database(20) / security(12) / monitoring(10) 等 12 个类别。形态上分三种：

对 coding agent 生态的意义

这个仓库回答的是生态最关键的问题：当任何人都能给 agent 写插件时，怎么保证安全与可信？ Anthropic 的答案是——标准化组件模型（skills/agents/hooks/MCP）+ 不可变 SHA 锁定（供应链完整性）+ LLM 策略扫描（行为安全）+ 全自动 bump/回滚（可维护性）。它为“AI 开发工具如何提供官方插件生态而不牺牲安全”立了一个可参考的范本。前两个项目本身，正是这类生态里最典型的“能力插件”——而这个仓库，定义了它们被分发和信任的规则。

"记忆" 这件事，被向量库做坏了

大多数 AI 记忆系统的结构是这样的：把每条消息切碎 → 跑 embedding → 塞进 pgvector / chroma / pinecone → 查询时 top-K 相似。这能回答"和我现在说的话最像的旧记忆是什么"，但回答不了真正的记忆问题。

OpenHuman 的 gitbook 直白地写：

"The Memory Tree is OpenHuman's knowledge base. It is not a vector database with a thin 'memory' wrapper. It is a deterministic, bucket-sealed pipeline that turns the messy stream of your day into structured, queryable, summary-backed Markdown that lives on your machine."

向量库失败在三件事上：

  • "今天发生了什么？" —— 这是一个全局摘要问题，没有 query 向量可以和它对齐。
  • "Alice 最近怎么样？" —— 这是一个实体路由 + 时间衰减问题，需要专门为某个实体维护视图。
  • "上周二下午 3 点那个 Stripe webhook 说了啥？" —— 这是一个来源 + 时间精确定位问题，需要溯源能力。

Memory Tree 给出的答案是：把"压缩"和"导航"同时做掉。embedding 仍然存在用于语义搜索，但结构层是真正让记忆"像大脑而不是碎片袋"的东西。下面我们逐层拆开这条流水线。

第一层 · 热路径

Canonicalize：把异构源压成同一种 Markdown

src/openhuman/memory/tree/canonicalize/ 下每种源一个适配器，全部产出同一个 CanonicalisedSource { markdown, metadata }。各适配器的处理细节很克制——拒绝在这一层做"理解"：

  • chat.rs：消息按时间戳防御性排序，每条以 ## YYYY-MM-DDTHH:MM:SSZ — User 起头；空 batch 直接返回 Ok(None) 让上层跳过。
  • email.rs：每封邮件之间用 ---\nFrom: 分隔；正文先过 email_clean::clean_body() 把引用历史、退订链接、营销页脚剥掉——给后面的摘要器一个信号干净的输入。
  • document.rs：本身已经是 Markdown 或纯文本，加一点 front-matter 就送下游。

Chunker：内容寻址的确定性 ID

这是整个系统能"幂等"的根因。chunk 的 ID 不是自增的，而是对内容求 SHA-256 取前 32 位十六进制。同一段输入永远算出同一个 ID，重复 ingest 直接 upsert，从源头杜绝重复。

src/openhuman/memory/tree/types.rs — 确定性 chunk_id
pub fn chunk_id(
    source_kind: SourceKind,
    source_id: &str,
    seq_in_source: u32,
    content: &str,
) -> String {
    let mut hasher = Sha256::new();
    hasher.update(source_kind.as_str().as_bytes());
    hasher.update([0u8]);
    hasher.update(source_id.as_bytes());
    hasher.update([0u8]);
    hasher.update(seq_in_source.to_be_bytes());
    hasher.update([0u8]);
    hasher.update(content.as_bytes());
    // → SHA256 取前 32 位 hex（128-bit，单用户 workspace 抗碰撞绰绰有余）
}

注意 content 也参与了哈希——这就允许同一来源在不同时间窗内有相同 seq_in_source 而仍然产生唯一 ID（比如 Slack 每 6 小时一桶，每桶 seq=0 起，但内容不同所以 ID 不同）。这是个不张扬但关键的设计取舍。

Content Store：vault 里的每个 .md 文件都是一笔"凭证"

chunk 写盘有两层冗余：SQLite 存索引和截断后的 content（≤2000 字符），原文以 .md 形式存到 wiki/raw/，YAML frontmatter + 正文。写入用 atomic::write_if_new()——先写临时文件、fsync、再 rename——保证就算进程崩在中间也不会留下半截文件。读回时还要核对 SHA256 防止 vault 被外部工具篡改。

SQLite 持久化：八张表撑起整棵树

所有结构化状态压在一个 SQLite 文件里：<workspace>/memory_tree/chunks.db，用 rusqlite，WAL + busy_timeout = 15s 让 ingest / worker / scheduler / scheduler 并发写不打架。核心表：

  • mem_tree_chunks —— 主表，128-bit ID + 来源/时间/owner 索引。
  • mem_tree_chunk_embeddings —— 按 model_signature 存向量 BLOB，支持多模型并存方便迁移。
  • mem_tree_score —— 每条 chunk 的全部信号分项（token / unique words / metadata / source / interaction / entity_density），可审计。
  • mem_tree_entity_index —— 实体倒排表，(entity_id, node_id) 主键，节点跨 chunk 与 summary。
  • mem_tree_trees / mem_tree_summaries / mem_tree_buffers —— 三棵树的状态机。
  • mem_tree_hotness —— 实体热度分（驱动 topic tree 物化）。
  • mem_tree_jobs —— 持久化作业队列（下一节展开）。

Scoring：两段式打分 + 准入门

分数控制"这条 chunk 配不配进树"。设计上分两段：快速路径在 ingest 同步阶段就跑，只用 token 数、词汇多样性这种纯启发式；慢速路径在 worker 上跑，做 LLM 实体抽取与 embedding。两个阈值把分数切成三段：

src/openhuman/memory/tree/score/mod.rs — 准入决策
pub struct ScoringConfig {
    // ...
    pub drop_threshold: f32,                // 默认 0.30
    pub definite_keep_threshold: f32,     // 默认 0.85 — 这分以上免 LLM 直接收
    pub definite_drop_threshold: f32,     // 默认 0.15 — 这分以下免 LLM 直接丢
}

// 决策伪码：
// fast >= 0.85   → admit                    (no LLM)
// fast <= 0.15   → drop                     (no LLM)
// 0.15 .. 0.85   → 调 LLM 实体抽取做二次判定 (borderline 才烧 token)

这个设计意图明确：用启发式吃掉两端的"显然要 / 显然不要"，把宝贵的 LLM 调用集中在中间灰区。准入失败的 chunk 不会被删——mem_tree_score.dropped = 1 记着，原文还在 vault 里，只是不会再被树或检索引用。这保留了完整溯源能力。

第二层 · 系统之心

三棵树，三种问题

Memory Tree 的招牌不是"一棵更复杂的树"，而是三棵独立、互不干扰的树同时生长，每棵回答一类不同的检索需求。它们都吃同一个 leaf 流，但封口规则、标签策略、时间窗各不相同。

SOURCE
源树 · per-source 滚动缓冲

每个 Slack 频道 / Gmail label / 文档一棵。新 leaf 进 L0 buffer，攒够就封成 L1 摘要，继续往上级联。

  • L0 封口：50k token 或 10 个 sibling
  • L1+ 封口：10 个 sibling（按数不按 token）
  • 过期强封：7 天没动也封
TOPIC
主题树 · 按"热度"懒物化

每个高热度实体（人、项目、ticker、repo）一棵。冷实体连树都不建；热度越高，刷新越积极。

  • 热度 = ln(30d 提及数) + 跨源数 + 近期衰减 + 中心度 + 用户查询权重
  • 由 topic_route 作业按门槛触发建树
  • 结构和源树相同，schema 共用
GLOBAL
全局树 · 每日一节点

单例。每天 UTC 零点后调度器跑一次，把昨天跨所有来源的最佳 L1+ 摘要合成成一节日报。

  • L0 = 日报；攒够 10 个升级 L1（周）
  • L1 升 L2（月）、L3（年）逐级展开
  • 跑出来的标签是 UnionFromChildren，免再调一次 LLM

桶封 (bucket-sealing) 的关键常量

tree_source/types.rs 直接把这套规则写成常量：

src/openhuman/memory/tree/tree_source/types.rs:188
pub const INPUT_TOKEN_BUDGET: u32 = 50_000;
pub const OUTPUT_TOKEN_BUDGET: u32 = 5_000;
pub const SUMMARY_FANOUT: u32 = 10;
pub const DEFAULT_FLUSH_AGE_SECS: i64 = 7 * 24 * 60 * 60;

// should_seal: L0 看 token 预算，L1+ 看兄弟数
pub fn should_seal(buf: &Buffer) -> bool {
    if buf.level == 0 {
        buf.token_sum >= INPUT_TOKEN_BUDGET as i64
            || buf.item_ids.len() as u32 >= SUMMARY_FANOUT
    } else {
        buf.item_ids.len() as u32 >= SUMMARY_FANOUT
    }
}

为什么 L0 按 token、L1+ 按数？一个非常工程师的考虑：summarizer 的输出长度有时长有时短。如果 L1 也按 token 封，万一摘要器吐得很短，L1 buffer 就要凑很多条才达到 50k token，造成"长链条窄树"。改成按 10 个 sibling 封口能保证树形结构稳定地"宽展开"——一棵看起来像树的树，而不是退化的链表。

级联封口（cascade seal）：当 L0 封成 L1，会立刻检查 L1 buffer 是否也到达门槛；到了就再封一层。一次 50k token 的注入可能一路级联到 L3。这套递归在 bucket_seal.rs::cascade_seals() 里完成，全程在一个 SQLite 事务内。

主题树的热度公式：用对数压住头部、奖励跨源

这是 tree_topic/hotness.rs:32 里非常值得品的设计——它在五个因子上做加权和，每个都有明确的物理意义：

src/openhuman/memory/tree/tree_topic/hotness.rs
pub fn hotness(entity_id: &str, idx: &EntityIndexStats) -> f32 {
    let mention = ((idx.mention_count_30d as f32) + 1.0).ln();     // log 压头部
    let source  = (idx.distinct_sources as f32) * 0.5;           // 跨源加分
    let recency = recency_decay(idx.last_seen_ms, now_ms);     // 分段线性衰减
    let central = idx.graph_centrality.unwrap_or(0.0);           // 图中心度 (Phase 4+)
    let query   = (idx.query_hits_30d as f32) * 2.0;             // 用户问了几次
    mention + source + recency + central + query
}

recency_decay 是分段线性的：≤1 天 = 1.0，1–7 天线性降到 0.5，7–30 天降到 0，再老不计入。这避免了指数衰减把"两个月前但很重要"的实体直接清零，又让 30 天的天花板硬性截断。query 权重 ×2 表达了一个朴素但正确的偏好：你查过的，比别人提过的更值得维护一棵树。

标签策略：三种打标方式

摘要节点封口时还要决定"这一节摘要的标签从哪来"。LabelStrategy 给了三个选项：

  • ExtractFromContent：对刚生成的摘要文本再跑一次实体抽取——能捕捉到"几条 leaf 各说一点、合起来才浮现的主题"。源树、主题树用这个。
  • UnionFromChildren：把子节点已有的实体/topic 去重并取并集——免一次 LLM 调用。全局树用这个，因为它的子节点本身就是已经被 L1 抽取过的源摘要。
  • Empty：留空，留给后续阶段填，防止跨树污染。
第三层 · 作业系统

慢活全部丢给队列，UI 永不阻塞

ingest 在热路径上只做"切碎、快速打分、入库、入队"。所有真正烧资源的事情——embedding、实体抽取、摘要生成、级联封口、日报合成——都被推进一个持久化的 SQLite 作业队列，由后台 worker 池跑。

六种作业类型

Kind做什么同步 / 异步调 LLM
extract_chunk深度打分 + 实体抽取；决定 admitted / dropped异步是
append_buffer把 admitted leaf 推进树的 L0 buffer；可能触发 seal异步否
seal把 buffer 压成一节摘要；视情况向上级联异步是（summarizer）
topic_route把 leaf 按实体路由进主题树（热度门控）异步是
digest_daily合成全局日报；满 10 天后级联到周/月节点异步是（summarizer）
flush_stale巡视 L0 buffer，强制封口超过 7 天没动的异步否

Workers：三个协程 + 一个 LLM 信号量

默认 3 个 worker 协程，循环从 mem_tree_jobs 表里抢 status='ready' 且 available_at_ms <= now 的作业。一个细节非常用心：所有标记 is_llm_bound()=true 的作业，在执行前必须从一个全局信号量获取许可——这把 LLM 并发调用上限锁住，防止"一次性接入 10 个 Slack 频道"把 embedding API 打到 rate-limited。

领到作业后，worker 立刻把行标记成 running，写入自己的 worker_id 与 lease_until_ms。如果它崩在中间，下一轮启动时 recover_stale_locks() 会扫到 lease 已过期的行，把它们重置回 ready 让别的 worker 重做。

失败处理：指数退避 + 显式 Defer

作业返回三种结果：Done 完成；Defer { until_ms, reason } 不消耗重试预算地延后（用于 LLM 限流这种瞬时不可用）；Err 真失败——按 2^attempts 秒指数退避丢回队列；超过 max_attempts（默认 5）就标 failed 进死信。

Scheduler：唯一的"墙钟"心跳

日历驱动的工作（日报、周强封）由一个独立的 scheduler 协程发起。它每天 UTC 大约 00:05 醒一次，向队列推两件事：

  • DigestDaily(yesterday) —— 合成昨天的全局日报
  • FlushStale(today) —— 检查所有 L0 buffer 中年龄 ≥7 天的，强制封口

scheduler 自己绝不执行 summarizer——它只发作业，让重试、去重、stale-lock 恢复都从作业队列统一走。这是个非常老练的"职责分离"设计。

Dedupe key 防重发：mem_tree_jobs.dedupe_key 是 UNIQUE 列，"digest_daily:2026-04-21" 这种键的第二次插入会直接被 SQLite 约束挡掉。即使 scheduler 重启两次，也只会有一份日报作业排队。
第四层 · 出口

六个检索原语，覆盖所有问法

树建得再好，对 agent 没用就是白搭。OpenHuman 把检索拆成六个明确的入口，每个回答一类不同的问题，统一返回 RetrievalHit——它把 leaf 和 summary 包装成同一种壳，agent 不用关心命中的是原始 chunk 还是某层摘要。

📡
query_source

在某个源树里搜（或不指定源、跨所有源）。支持时间窗 + 自然语言 query；无 query 时按时间倒序。

👤
query_topic

给一个实体 ID（人/项目/repo），把它的主题树过一遍。这是"Alice 最近在干嘛"的标准入口。

🌍

查全局树，配合日期参数取"昨天" / "上周" / "本月" 的整体摘要。

🔬

给一个 summary 节点 ID，把它的 child_ids 全部展开——一路下钻到原始 chunk + provenance。

🎯

按 ID 精确取节点（leaf 或 summary）。给 UI 跳转 / IDE 链接用。

🩺

不是检索而是监控：每个 provider 已同步 / 待处理 chunk 数 + "freshness"（active/recent/idle）。

混合排序：结构信号 + 语义相似 + 时间衰减 + 实体匹配

有 query 时走 hybrid：embedding 算余弦相似 → 与 BM25-ish 结构分加权 → 加入 recency → 如果 query 里包含已知实体名，命中实体的节点再加权。无 query 时则纯按时间和结构分排序，这让"列出最近的"和"语义搜索"是同一个入口的两种用法。

溯源：每个 hit 都带 provenance

命中里有 source_ref（永久指向 Slack 消息 / Gmail 线程 / 文档段落的不透明指针），UI 上点 hit 就跳回 Obsidian vault 的原文件。这一点彻底解决了"AI 模糊地引用一句话但说不出哪来"的问题——所有摘要都能被打开还原成它的全部子节点和原始 chunk。

Agent 怎么用：DefaultMemoryLoader 注入

agent 端的 memory_loader.rs 在每次用户发消息时调 memory.recall(user_msg) 取相关条目，然后把它们组织成四个段落注入 system prompt：

  • [Memory context] —— 语义召回的相关记忆
  • [User working memory] —— ≤3 条用户画像（时区、偏好）
  • [Cross-chat context] —— 其他对话线程里的近期命中
  • [Prior conversations] —— 之前会话里的高优先 episodic 条目

SessionMemoryState：用三个增量决定何时抽取

context/session_memory.rs 跟踪当前会话的累计 token、tool 调用数、turn 数。只有这三个增量同时跨过门槛（token +4000、tool +8、turn +4）才会触发后台的"档案管理员"子代理把会话压成结构化记忆条目——避免每隔几句话就启一次抽取作业。

护栏

本地存储，但不等于裸奔

所有数据都不出机器，但"不出机器"不代表"可以随便存"。memory/safety/mod.rs 在写入与索引时就做了四类拦截，故意倾向于宁可误删也别留下密钥的"保守派"策略。

  • Block 模式（直接删）：PEM 私钥（RSA / EC / OpenSSH / PGP 各变体），高熵 base64 段。
  • Redact 模式（替换为 [REDACTED]）：Bearer token、AWS AKIA…、GitHub gh*_、Anthropic sk-ant-…、Google AIza…、OpenAI / Stripe sk|rk_live|test_…、Slack xox[abprs]-…、JWT、OAuth。
  • Key Redaction：JSON 里 "api_key" / "token" / "password" / "secret" 这种键的值，直接擦掉——不管值长啥样。
  • Depth Limit：深嵌套 JSON 做截断防 DoS。

这套清洗发生在三个时机：写入 chunk 前、抽取实体前、把记忆切片写进日志前。"宁可误擦"的取向意味着偶尔会把无害字符串当成 token 扔掉，但绝不会让一个真 token 进入摘要再被某天的检索意外暴露给 LLM。

合而观之

为什么不是又一个向量库

把所有细节合起来看，Memory Tree 与"套壳向量库"差的不是某一个特性，而是把记忆当作什么的根本分歧。

问题向量库Memory Tree
今天发生了什么？无解（没有 query 向量）全局树昨日 L0 节点
Alice 最近怎么样？top-K 相似，可能漏掉关键她的 topic tree，热度自动维护
上周二 3 点 Stripe 说了什么？需要全文 + 时间过滤补打源树 + time range 直接命中
这条摘要从哪来？top-K 命中没有溯源链summary.child_ids → 原 chunk → vault .md
重新 ingest 相同内容？会产生重复向量SHA-256 chunk ID 直接 upsert
能 git 化、和团队共享？难（向量是 blob）vault 就是 Markdown，可 commit
断电恢复？看实现持久化 job 队列 + lease 过期回收
LLM 用量？每次查询都嵌入双阈值切灰区，只在 0.15–0.85 调 LLM
核心立场：向量库回答 "和这句话最像的旧记忆是什么"；Memory Tree 回答 "今天/这个人/这个源最重要的事情是什么，并能把它一路展开成原文"。前者是相似性检索，后者是结构化记忆。两者的输入相似，但输出完全不在同一类问题里。

可选后端：兼容 agentmemory

值得一提的是 OpenHuman 没有把自己锁死。MemoryConfig.backend = "agentmemory" 可以让它把存储层换成另一个独立项目 agentmemory 的 REST 服务——这样多个 coding agent（Claude Code、Cursor、Codex、OpenCode、OpenHuman）可以共享同一个持久化记忆。Memory Tree 流水线本身仍在跑，但底层 Memory trait 的存储被换掉。这种"流水线 + 后端"解耦的态度，比大多数声称要做生态的项目都更脚踏实地。

对个人 AI 的意义

Memory Tree 的真正价值在两件事上。第一，它示范了"确定性流水线 + LLM 重活异步化"这种组合拳——把可以确定的部分（切分、哈希、入库、调度）做得像数据库一样稳，把 LLM 调用约束在借灰区判断、摘要生成这种能容忍重试和失败的位置。第二，它把"个人 AI 的记忆"从云服务的私有 schema 抢回了用户的硬盘——你可以打开 ~/.openhuman/wiki/ 直接用 Obsidian 读，可以 git 化、可以备份、可以审计、可以删。本地优先不是宣传话术，而是体现在每一个文件路径里。

一句话定位：如果说前四个项目是"让 agent 更懂代码、更有能力、更有记忆"，那 Kimi Code 的回答是——如何让多个 agent 像一个工程团队一样协作。它的核心是一个多会话并行框架：一个"主调度 agent"负责理解需求、拆解任务，多个"工作 agent"并行执行，最后由主 agent 整合结果。全部跑在本地，以 Python + Vue 实现。

多 Agent 协作架构

Kimi Code 的设计哲学是"会话即团队"——每个用户会话对应一个主 agent，主 agent 维护一个任务队列，按需派生出多个工作 agent（类似"子线程"），每个工作 agent 有自己的上下文窗口和工具集，主 agent 负责任务分发、结果汇总、状态协调。这种模式在工程上的好处是：解耦——不同 agent 处理不同模块，主 agent 只需要维护任务列表而不陷入具体实现。

为什么不用单个更强的 agent？Kimi Code 的核心假设是：复杂任务需要多视角并行处理。一个 agent 顺序处理多个模块时，前面模块的上下文会"稀释"后续模块的处理质量；而多 agent 并发意味着每个 agent 都能在新鲜、热乎的上下文中工作，主 agent 最后负责"拼图"。

代码库索引：语义 + 结构双引擎

Kimi Code 继承了 Kimi 在长上下文上的优势，同时在本地增加了结构化索引层——它会先用 tree-sitter 解析代码结构（函数、类、依赖），建立轻量级的代码图谱；再对函数体做 embedding，用于语义搜索。这样"找某个功能"走语义向量，"理解模块关系"走结构图谱，两条路互不干扰。

记忆层：会话间的持久化上下文

区别于 OpenHuman 的"个人 AI 记忆"，Kimi Code 的记忆层更侧重项目级上下文——它会记住项目里有哪些模块、哪些是核心类、哪些文件是入口点。新会话进来不需要重新探索，可以直接继承这些"项目记忆"，让 agent 在新任务中立刻进入状态。

工程化：开箱即用的本地部署

Kimi Code 强调"零配置体验"——pip install 后一个命令启动，自动检测本地模型（支持 Ollama / vLLM / Kimi API），Web UI 内置多会话管理、agent 状态监控、代码编辑器预览。对于中小团队而言，这意味着不需要懂得 agent 编排，就能用上多 agent 协作的编码体验。

对 coding agent 生态的意义

Kimi Code 的价值在于降低了多 agent 协作的工程门槛。它示范了一种"以会话为中心"的多 agent 模型——不是让一个 agent 变得越来越强，而是让多个各有专精的 agent 通过一个共享上下文协作。它的记忆层和索引层设计，也为"如何在本地跑一个真正好用的 coding agent"提供了工程答案。随着 coding agent 逐渐从"新鲜玩具"变成"日常工具"，这种开箱即用、可私有、可扩展的方案，会比"云端 API"更有生命力。
