一、项目定位

Spec Kit（仓库 github/spec-kit）是 GitHub 官方推出的开源工具包，用于推行 规格驱动开发（Spec-Driven Development, SDD）。 它的核心主张是：把"规格说明"从一次性、可丢弃的脚手架，反转为可执行的源头工件——规格直接驱动 AI Agent 生成实现，代码反而成为规格的"衍生物"。

口号："build high-quality software faster" —— 先聚焦"做什么（What）"再决定"怎么做（How）"，把 AI 编程从"一次性 prompt 生成"升级为"多步精炼、有组织护栏的可控产出"。

截至 2026-05，仓库约 98K stars / 8.5K forks，最新版 v0.8.9（2026-05-12），144+ releases，节奏极为活跃。项目灵感与基础研究来自 John Lam（jflam）。

定位关键词： specify CLI · 7 步 SDD 工作流 · 30+ AI Agent 覆盖 · 120+ 社区 Extensions · 4 级模板优先级解析 · 企业级可追溯。

二、什么是规格驱动开发（SDD）

传统开发把规格当作写完即弃的草稿；SDD 翻转了这个关系。其核心理念：

  • 意图驱动（intent-driven）——先回答 What，再决定 How；
  • 富规格 + 组织护栏——规格不只是需求，还携带项目宪章级别的约束；
  • 多步精炼优于一次性生成——把"一个大 prompt 生成全部"拆成宪章→规格→澄清→计划→任务→实施的链路；
  • 充分利用先进模型解释规格的能力——让 Agent 在每一步都有明确、可审计的输入产物。

SDD 的本质，是给"AI 写代码"这件事装上工程流程的护栏与可追溯的工件链。

三、七步 SDD 工作流

Spec Kit 把开发拆成 7 个阶段，每个阶段对应一个 /speckit.* 斜杠命令，产出一个明确的 Markdown 工件：

步骤命令产出工件要点
1/speckit.constitution.specify/memory/constitution.md制定代码质量、测试、UX、性能等治理原则
2/speckit.specifyspecs/[feature]/spec.md只谈需求与用户故事，不谈技术栈
3/speckit.clarifyspec.md 的 Clarifications 段结构化顺序追问，消灭"灰色地带"，减少返工
4/speckit.planplan.md + contracts/ + data-model.md + research.md选定技术栈、画契约、定数据模型
5（Plan 审计）—让 Agent 自审计划是否完整、是否过度设计
6/speckit.taskstasks.md生成有依赖顺序、带 [P] 并行标记的任务清单
7/speckit.implement实际代码按 TDD 逐项实施，尊重任务依赖

另有 3 个可选增强命令：/speckit.analyze（跨工件一致性分析）、/speckit.checklist（定制 QA 检查表）、/speckit.taskstoissues（任务转 GitHub Issues）。

四、扩展生态架构

Spec Kit 不是一个封闭工具，而是一个带优先级解析的可扩展平台。

flowchart TB SK(["github/spec-kit · specify CLI · Python 3.11+"]) SK --> Core["核心层"] SK --> Ext["120+ 社区 Extensions"] SK --> Pre["社区 Presets"] SK --> Agents["30+ AI 编码 Agent"] Core --> C1["7 个核心斜杠命令"] Core --> C2["模板系统 · 4 级优先级解析"] Core --> C3["constitution.md 项目宪章"] Ext --> X1["Architecture Guard"] Ext --> X2["Jira Integration"] Ext --> X3["Cost Tracker"] Ext --> X4["Security Review"] Ext --> X5["Bugfix Workflow"] Pre --> P1["Agile / Kanban / DDD"] Pre --> P2["Security Gate"] Pre --> P3["本地化 / 合规追溯"] Pre --> P4["Test-First 顺序"] Agents --> A1["Claude Code"] Agents --> A2["GitHub Copilot"] Agents --> A3["Gemini CLI / Cursor / Qwen / Codex / opencode"]

Extension vs Preset：何时用哪个

目标用
新增全新命令 / 工作流Extension
定制 spec/plan/task 的格式与术语Preset
集成外部工具 / 服务（Jira、成本追踪）Extension
强制组织标准（安全门禁、合规追溯）Preset
关键判断： Extension 是"加能力"，Preset 是"改行为"。Preset 不增加新命令，只覆盖模板与术语；多个 Preset 可按优先级叠加。

五、4 级模板优先级解析

Spec Kit 的模板在运行时解析，命令在 init 时安装。解析按以下优先级（高→低）：

1. 项目本地覆盖   .specify/templates/overrides/     ← 最高优先级
2. Presets        .specify/presets/templates/
3. Extensions     .specify/extensions/templates/
4. 核心模板        .specify/templates/                ← 最低优先级

# 移除 Preset / Extension 会自动回落到下一优先级版本

这套设计的好处：团队可以在不 fork 上游的前提下，用 overrides/ 做项目级定制，同时享受上游核心模板的更新。

六、快速上手

环境要求：Python 3.11+ · Git · uv 或 pipx · 一个受支持的 AI 编码 Agent。

# 持久化安装 (推荐, 用 uv; 建议 pin 版本 tag)
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@v0.8.9

# 一次性运行 (不安装)
uvx --from git+https://github.com/github/spec-kit.git specify init my-project

# 初始化项目
specify init my-awesome-app && cd my-awesome-app

# 查看支持的 Agent / 管理扩展与预设
specify integration list
specify extension search   &&  specify extension add architecture-guard
specify preset search      &&  specify preset add security-first

企业 / 内网隔离环境：仓库提供 pip download 打包可移植 wheel 的安装指南（见 docs/installation.md）。

在 Claude Code 中跑完整工作流

/speckit.constitution      # 1. 项目宪章
/speckit.specify           # 2. 编写规格
/speckit.clarify           # 3. 结构化澄清
/speckit.plan              # 4. 技术方案
/speckit.tasks             # 5. 拆任务
/speckit.implement         # 6. 实施
/speckit.taskstoissues     # 7. (可选) 转 GitHub Issues

七、三大使用场景

场景说明
0→1 开发（Greenfield）从高层需求一路生成可工作的应用——规格、计划、实施步骤全链路
创意探索（Creative Exploration）并行尝试不同技术栈、不同 UX 的多个实现，对比择优
迭代增强（Brownfield）往遗留系统增量加功能、做现代化改造，让 AI 在既有约束下工作

项目的实验性研究目标还包括：技术独立性（SDD 跨技术栈适用性）、企业约束下的关键任务开发、面向不同用户群体的开发偏好适配。

优势

  • GitHub 官方背书，institutional pull 强，已被 Claude Code / Copilot 文档引为推荐用法
  • 7 步工作流是真正的全闭环方法论，不是零散技巧
  • 4 级模板解析让团队定制与上游更新可以共存
  • 30+ Agent 覆盖，不绑定单一工具

风险 / 短板

  • 流程偏重，小型/探索性任务套用全流程会有"仪式感开销"
  • 274+ open issues / 134+ open PR，快速迭代意味着 API 仍在变动，建议 pin 版本
  • 扩展目录只校验完整性与格式，不审查代码质量与安全
  • 效果强依赖底层模型解释规格的能力
