一、项目定位与核心命题

OpenHuman（仓库 tinyhumansai/openhuman）是一款本地优先（local-first）的"个人 AI 超级智能"桌面助手。 它要解决的不是"再做一个聊天框"，而是一个更尖锐的工程命题：如何让一个 AI 助手在普通笔记本上，拥有近乎无限、可导航、低成本的长期记忆。

官方一句话定位："Your Personal AI super intelligence. Private, Simple and extremely powerful." 作者明确把它归类为 ANI（Artificial Narrow Intelligence，人工窄智能），而非 AGI——这是一个克制且诚实的定位。

与多数"云端 Agent + 向量库"的范式不同，OpenHuman 的三大设计支柱是：

  • Memory Tree——用"确定性管道 + 分层摘要树"取代纯向量库，让记忆可导航、可溯源；
  • Neocortex / Subconscious Loop——号称可处理 10 亿 token 级记忆、且不依赖 LLM 做索引的记忆引擎，并模拟"潜意识回忆"；
  • TokenJuice——在数据进入 LLM 前做确定性压缩，把成本与延迟最多砍掉 80%。
事实校准： 本报告分析的是 URL 对应的真实项目 tinyhumansai/openhuman（Rust + Tauri 桌面 AI 助手）。 它不是 Open-Sora 视频生成模型，因此不涉及 DiT / VAE / Tokenizer / CogVideoX 等视频扩散技术栈。 若你需要的是 Open-Sora 分析，请告知，我会另起一份。

二、核心架构总览

OpenHuman 是一个 Tauri 桌面壳 + Rust 核心 + TypeScript UI 的 monorepo。Rust 占 ~69%，承担记忆管线、工具调度、压缩等"重活"；TypeScript 占 ~26%，负责 UI 与编排。

为什么用 Tauri 而非 Electron

Tauri 用系统 WebView + Rust 后端，包体小、内存低，且能直接复用 Rust 写的记忆/压缩管线——避免了 Electron "JS 全栈但性能受限" 的取舍。

为什么 Rust 占比这么高

记忆管线的热路径（分块、打分、原子持久化）和 TokenJuice 压缩都是 CPU 密集、需要确定性的活，Rust 的零成本抽象与可预测性能在这里是刚需。

三、Memory Tree 深度解析（核心创新）

Memory Tree 是 OpenHuman 最有工程含量的部分。它不是向量库，而是一个"确定性、桶封存（bucket-sealed）的管道"，把一天里杂乱的输入流（聊天、邮件、文档、集成同步结果）变成结构化、可查询、带摘要的 Markdown，全部存在本机。

3.1 处理管线（热路径 vs 后台）

设计的精髓在于把慢的事（LLM 调用、embedding、摘要树构建）全部推到后台，热路径只做确定性的廉价操作，保证 UI 永不阻塞。

3.2 三种并存的摘要树

树类型作用域构建策略能回答的问题
Source Tree
源树		每个独立源（Gmail 标签 / Slack 频道 / 上传文档）滚动缓冲，多级 L0→L1→L2…，缓冲满则封存压缩"周二下午3点那个 Webhook 说了什么"——精确溯源
Topic Tree
主题树		高频实体（人、项目、ticker、仓库）惰性物化，实体出现越频繁，主题树构建/刷新越激进（hotness 驱动）"这个人最近的进展是什么"
Global Tree
全局树		每个 UTC 自然日的全部内容单一日摘要，随天数累积分层增长"今天发生了什么"

3.3 Chunk 生命周期状态机

stateDiagram-v2 [*] --> pending_extraction: 热路径写入 pending_extraction --> admitted: extract_chunk 深度打分通过 pending_extraction --> dropped: 打分未通过 admitted --> buffered: append_buffer 进入 L0 buffered --> sealed: seal 级联压缩 dropped --> [*]: 保留供溯源,不影响摘要 sealed --> [*]: 终态,保留溯源

双阶段打分是关键设计：Fast-Score（热路径，廉价启发式，无 LLM）只做粗筛并标记；Deep Score（后台 extract_chunk 作业）才做完整打分 + 实体抽取，决定 chunk 是 admitted 还是 dropped。一个共享信号量（semaphore）限制并发的 LLM 操作，防止突发摄入时意外扇出。

为什么是树而不是纯向量库

向量库擅长"语义相似"，但答不了"今天发生了什么"（需要时间维度）、"这个人最新进展"（需要 hotness 聚合）、"周二3点那条消息"（需要精确溯源）。 树结构同时提供导航层级 + 压缩 + 溯源，embedding 仍保留用于语义检索——这是"树为主、向量为辅"的混合策略。

四、Neocortex 与 Subconscious Loop

作者在 DEV.to 的技术博客里把记忆引擎称为 Neocortex，并提出了几个相当激进的指标声明：

声明数值
记忆容量可"准确工作"于 10 亿（1B）token 以上
索引速度10 分钟级 → 10M token / <10 秒（号称比替代方案快 1000×）
硬件要求CPU 级硬件即可（MacBook Air 可跑）
成本索引 5M token 约 $1
索引方式不使用 LLM，依赖实体理解 + 时间理解
排序因子按时间、交互次数、随机性因子排序

4.1 Subconscious Loop（潜意识循环）

设计灵感来自人脑的 Purkinje 细胞神经元。实现上：

  • Neocortex 周期性触发"核心记忆回忆"；
  • 每天运行 10,000+ 次，总成本 <$1；
  • 从回忆出的记忆中生成自主动作或确认，模拟人类"意识/潜意识"交互。
研判： 这些指标目前主要来自作者自述博客，缺少独立第三方基准。"1B token / 不用 LLM 索引 / CPU 可跑" 在工程上意味着它本质更接近一个带实体与时间索引的确定性检索引擎，而非语义模型——这与 Memory Tree "树为主、向量为辅" 的定位是自洽的。读者应把它当作有前景但待验证的设计。
作者引用 CMU 的研究 "Forgetting Is a Feature, Not a Bug"——主张选择性记忆比"全量塞上下文"更能提升准确率。这正是 Memory Tree 会 drop 低分 chunk 的理论依据。

五、TokenJuice 压缩管线

TokenJuice 是一个确定性压缩层：每一次工具调用、抓取结果、邮件正文、搜索负载，在抵达 LLM 之前都会先过一遍压缩。

压缩手段

  • HTML → Markdown（剥离标签噪声）
  • 长 URL 缩短
  • 去除非 ASCII 字符
  • 结构化裁剪——"同样的信息，零头的 token"

效果：成本与延迟最多降低 80%。它和 Memory Tree 是互补的——Memory Tree 解决"记什么、怎么导航"，TokenJuice 解决"喂给 LLM 的每一段都尽量便宜"。

与 Karpathy "LLM Wiki" 模式的关系

OpenHuman 的 Memory Tree 思路与 Andrej Karpathy 提出的"LLM 个人知识库（LLM Wiki）"一脉相承——都主张把知识沉淀为人可读、可编辑的 Markdown，而非锁进不透明的向量库。OpenHuman 的差异化是把它产品化、自动化、并加上确定性管道与三种摘要树。

优劣势研判

优势

  • 本地优先 + 全开源，隐私叙事扎实
  • Memory Tree 是真正有工程含量的差异化
  • 热路径/后台分离的架构成熟、UI 不阻塞
  • 118+ 集成构成强护城河

风险/短板

  • Neocortex "1B token" 等指标缺第三方验证
  • 仍是 Early Beta（v0.53.x），发布极频繁意味着不稳定
  • 桌面吉祥物/会议替身偏"产品噱头"，对严肃用户价值待证
  • GPL-3.0 对商业集成方有传染性约束
