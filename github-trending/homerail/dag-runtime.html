DAG 运行时：把 agent handoff 变成可恢复状态机

这一页只看运行时：DAGRun 的状态结构、handoff 路由、失败传播、ActiveRun 持久化和冷恢复。

状态模型：少字段，但含义很硬

READY 推进：after 依赖和显式输入必须同时满足

节点从 PENDING 到 READY 的关键不只是所有 after_dep 完成；如果某个依赖还有显式 data edge，目标节点还必须真的收到 routed input。这个双条件写在 sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:107、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:114、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:121、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:126。未被选择的分支会在依赖都满足但缺少输入时被 SKIPPED，逻辑位于 sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:129 到 sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:148。

Handoff：端口匹配决定工作流下一跳

handoff 会过滤同 source node、非 after_dep、非 terminal 且端口匹配的 downstream edge；terminal edge 单独处理，并用 failure port 与 terminal_outcome 决定 COMPLETED/FAILED/CANCELLED/RUNNING。证据在 sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:260、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:270、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:277、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:280、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:284、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:296。写入 mailbox 和 inputSatisfied 的代码在 sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:298 到 sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:303。

 294    );

 299      _ensureMailbox(run, edge.to_node, edge.to_port).push(content);
 300      run.inputSatisfied.get(edge.to_node)?.add(fromNode);
 301      affected.add(edge.to_node);
 302      mailboxReceivers.add(edge.to_node);
 303    }

失败传播：on_failure 不是异常通道，而是合法边

failNode() 不只是把节点置 FAILED，它会满足 after dep、寻找 condition 为 on_failure 或 always 的边，把错误数据塞进目标 mailbox，再唤醒目标；没有替代路径的普通后继才会被跳过。相关实现集中在 sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:324、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:332、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:335、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:344、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:351，而依赖跳过和 alternative path 判断在 sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:169、sources/homerail/homerail_manager/src/orchestration/dag-engine.ts:185。

 325    run: DAGRun,
 326    nodeId: string,
 327    errorData: unknown = "",
 328  ): DAGTransitionResult {
 329    if (!run.nodeStates.has(nodeId)) {
 330      throw new Error(`Unknown node: ${nodeId}`);
 331    }
 332    run.nodeStates.set(nodeId, "FAILED");
 333    const affected = _satisfyAfterDeps(run, nodeId);
 334    const mailboxReceivers = new Set<string>();
 335    for (const edge of run.graph.edges) {
 336      if (edge.from_node !== nodeId) continue;
 337      if (edge.label === "after_dep" || edge.to_node === "") continue;
 338      if (edge.condition !== "on_failure" && edge.condition !== "always") continue;
 339      _ensureMailbox(run, edge.to_node, edge.to_port).push(errorData);
 340      run.inputSatisfied.get(edge.to_node)?.add(nodeId);
 341      affected.add(edge.to_node);
 342      mailboxReceivers.add(edge.to_node);
 343    }
 344    for (const affectedNode of affected) {
 345      _tryPromote(run, affectedNode);
 346      _skipUntakenSatisfiedBranches(run, affectedNode);
 347    }
 348    for (const receiver of mailboxReceivers) {
 349      _wakeLoopSource(run, receiver);
 350    }
 351    _skipDependentNodes(run, nodeId);
 352    return {
 353      affectedNodes: Array.from(affected).sort(),
 354      routedNodes: Array.from(mailboxReceivers).sort(),
 355    };

 462  //  Cold recovery — rebuild an ActiveRun from persisted state after a restart.
 463  //
 464  //  The persisted RunMetadata covers nearly every ActiveRun field, but four
 465  //  DAGRun Maps are process-local and NOT serialized (mailboxes, afterSatisfied,
 466  //  inputSatisfied, loopSources). Rather than hand-writing reconstruction and
 467  //  risking loop-gateway wake-ups or dependency-satisfaction edge cases, we
 468  //  rebuild a minimal DAGRun (all nodes PENDING, empty Maps) and replay the
 469  //  persisted dag_handoffs through the engine's own `handoff()` — which drives
 470  //  mailbox pushes, inputSatisfied marking, afterDep satisfaction, tryPromote
 471  //  and loop wake-ups exactly as it did the first time. The authoritative
 472  //  nodeStates snapshot from metadata is then layered on top so transient
 473  //  states (RUNNING) are preserved long enough to be demoted.
 474  // ---------------------------------------------------------------------------
